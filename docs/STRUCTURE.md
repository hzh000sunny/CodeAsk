# CodeAsk 文档目录结构与版本约定

> 本文是 CodeAsk 文档系统的**权威约定文件**。所有 PRD / SDD / 草稿的组织方式都以此为准。
>
> 本文不属于任何具体版本——它描述的是版本目录系统**自身**的规则。

---

## 1. 顶层目录结构

```text
docs/
├── README.md          ← 顶层索引：当前版本指针 + 历史版本列表
├── STRUCTURE.md       ← 本文件：版本约定的权威来源
├── v1.0/              ← 当前版本（示例）
│   ├── README.md      ← 该版本的元信息和导航
│   ├── prd/           ← 产品需求文档
│   ├── design/        ← 系统设计文档（SDD）
│   ├── plans/         ← 实现计划（拆 SDD → 可执行 task list）
│   └── specs/         ← 早期草稿 / 过程性产物
├── v1.1/              ← 未来 minor 演进
└── v2.0/              ← 未来 major 演进
```

**底层原则**：每个 `vN.M/` 是一次产品演进的**完整文档快照**。各版本之间互不干扰、可独立阅读。

---

## 2. 版本目录命名

采用**语义化版本**（Semantic Versioning）：

| 版本号格式 | 示例 | 含义 |
|---|---|---|
| `vMAJOR.MINOR` | `v1.0/`、`v1.1/`、`v2.0/` | 主版本 + 次版本，绝大部分情况用这个 |
| `vMAJOR.MINOR.PATCH` | `v1.0.1/` | 仅在需要保留某次 patch 的快照时使用，不强制 |

**版本号目录命名规则**：
- 全部小写 `v` 前缀
- 数字间用 `.` 分隔
- 不使用 `_`、`-` 或其他分隔符
- 不带日期或主题后缀（如 `v1.0-mvp/` 是**反例**）

---

## 3. 版本内子目录

每个版本目录下固定使用以下子目录：

| 子目录 | 内容 | 必须有吗 |
|---|---|---|
| `prd/` | 产品需求文档（PRD），描述"为谁 / 为什么 / 做什么 / 不做什么" | 是 |
| `design/` | 系统设计文档（SDD），描述"怎么实现"。多个主题文档并列 | 是 |
| `plans/` | 实现计划：把 SDD 拆成 bite-sized、TDD、可执行的 task list（一份计划一个文件，按主题命名，如 `foundation.md`、`wiki-knowledge.md`） | 可选 |
| `specs/` | 早期草稿、过程性产物、决策快照 | 可选 |

**禁止**：
- 子目录里再嵌版本号（如 `v1.0/prd-v1.0/` 是**反例**）
- 子目录里再分日期（如 `v1.0/design/2026-04-29/` 是**反例**）

子目录直接用语义化命名（`prd/`、`design/`、`specs/`），版本号已经在外层。

---

## 4. 文件命名

### 4.1 不使用日期前缀

文件名按**主题**命名，不带日期：

| ✓ 正例 | ✗ 反例 |
|---|---|
| `prd/codeask.md` | `prd/2026-04-29-codeask-product-prd.md` |
| `design/agent-runtime.md` | `design/2026-04-27-agent-runtime.md` |
| `specs/codeask-initial-draft.md` | `specs/2026-04-27-codeask-design.md` |

**原因**：版本目录已经承担了"时间快照"的语义，文件名再加日期就是噪音。何时创建的、改了什么时候——git 历史是权威来源。

### 4.2 使用小写 + 短横线

- 全部小写
- 多词用短横线分隔（kebab-case）
- 例：`agent-runtime.md`、`api-data-model.md`

### 4.3 README.md 例外

`README.md` 文件名沿用大写惯例（开源生态约定）。

## 5. 版本 bump 规则

### 5.1 三种改动类型

| 改动类型 | 触发 | 处理 |
|---|---|---|
| **MAJOR**（v1.0 → v2.0） | PRD 主链路推翻、画像换、产品定位改变 | **新建** `v2.0/` 目录（基于上一版复制后改） |
| **MINOR**（v1.0 → v1.1） | 显著新增功能 / 章节，但不推翻方向 | **新建** `v1.1/` 目录（基于上一版复制后改） |
| **PATCH** | typo / 表述优化 / 不影响理解的微调 | **就地改**，git 历史保留细节 |
| **PATCH 显式标记**（按需） | 需要保留某次 patch 快照（如发布给外部时） | 新建 `v1.0.1/` 目录（不强制，按需） |

### 5.2 怎么判断 MAJOR / MINOR / PATCH

| 改动会导致… | 走哪一档 |
|---|---|
| SDD 大量需要重写 / 主链路图要重画 | MAJOR |
| 新增一个产品契约 / 新增一节 / 改动若干 SDD 但主链路不动 | MINOR |
| 修个错别字 / 调一段表述 / 补一个例子 | PATCH（git 改） |
| 给某个 PATCH 留快照（如发布给客户） | PATCH 显式标记 |

经验法则：**能用 git diff 一两行说清楚 → PATCH（git 改）**；**改动覆盖多节 / 影响 SDD → 至少 MINOR（开新版）**。

### 5.3 旧版本目录的命运

**旧版本目录一律不就地覆盖。**

- 即使 v2.0 已经发布，v1.0 目录仍然保留
- 不删除、不归档到别处
- 这让历史阅读成为可能（"当时为什么这样设计"）

## 6. 引用路径风格

### 6.1 同版本内引用

文档之间引用**同版本内的别的文档**时，用 **不带 `docs/vN.M/` 前缀** 的同级目录路径：

| ✓ 正例 | ✗ 反例 |
|---|---|
| `design/overview.md` | `docs/v1.0/design/overview.md` |
| `specs/codeask-initial-draft.md` | `/Users/.../docs/v1.0/specs/codeask-initial-draft.md` |

**原因**：当从 v1.0 fork 出 v2.0（整目录 cp）时，所有内部引用自动有效，不需要逐文件改路径。

### 6.2 跨版本引用

引用**别的版本**的文档时，用**相对路径**：

| 场景 | 路径 |
|---|---|
| v2.0 的 PRD 引用 v1.0 的 PRD 作为"上一版对照" | `../v1.0/prd/codeask.md` |
| v1.1 引用 v1.0 的 design 文档 | `../v1.0/design/overview.md` |

### 6.3 引用顶层 docs/README 或 STRUCTURE

| 来源位置 | 路径 |
|---|---|
| 从 vN.M/ 内部 | `../README.md` 或 `../STRUCTURE.md` |
| 从 vN.M/prd/、design/、specs/ 内部 | `../../README.md` 或 `../../STRUCTURE.md` |

## 7. README.md 的层级和职责

### 7.1 顶层 `docs/README.md`

- 指向当前版本（如 `[v1.0](./v1.0/)`）
- 列出历史版本
- 给出 STRUCTURE.md 链接（详细约定）

**保持简短**——它是入口，不是百科。

### 7.2 版本级 `docs/vN.M/README.md`

- 该版本元信息（版本号、起始日期、状态、主题）
- 该版本目录结构
- 推荐阅读顺序
- 与上一版的对齐状态（如有）

### 7.3 PRD 内 "文档维护" 小节

- 简要复述本约定（便于 PRD 单独阅读）
- 引用本文 STRUCTURE.md 作权威来源

避免在多处独立维护版本规则——以 STRUCTURE.md 为准。

## 8. PRD vs SDD vs Plans vs Specs 关系

| 类别 | 位置 | 回答的问题 | 变更频率 | 冲突时谁赢 |
|---|---|---|---|---|
| PRD | `vN.M/prd/` | 为谁 / 为什么 / 做什么 / 不做什么 | 低（版本契约） | **赢** |
| SDD | `vN.M/design/*.md` | 怎么实现 | 中（实施学到新东西可改） | 应同步更新 |
| Plans | `vN.M/plans/*.md` | 怎么落地（具体 task / 测试 / 提交节奏） | 实施期间高，落地后冻结 | 落地以代码为准；与 SDD 冲突时改 plan |
| Specs | `vN.M/specs/` | 历史草稿、决策快照、过程产物 | 仅追加，不改写 | 仅供参考 |

**核心规则**：PRD 与 SDD 冲突时，以 PRD 为准。SDD 与 Plans 冲突时，以 SDD 为准。SDD 与 Specs 冲突时，以 SDD 为准。

**Plans 的边界**：Plans 是**前瞻**的实现路线（"接下来要做的 task"），不是事后的实施记录。Plan 落地、对应代码合并后，plan 文件本身**冻结不再改**——后续学到的事情体现在代码 + commit message + SDD 增补里，不回头改已发布的 plan。

## 9. 新建一个版本的标准流程

假设当前是 v1.0，要新建 v1.1：

```bash
# 1. 复制整个版本目录
cp -r docs/v1.0 docs/v1.1

# 2. 修改 docs/v1.1/README.md
#    - 版本号 → v1.1
#    - 起始日期 → 今天
#    - 状态 → Draft（待审阅）
#    - 主题 → 描述这次 minor 演进的核心变化

# 3. 在 v1.1/ 内做实质改动
#    - 改 PRD（标注本次变化的章节）
#    - 同步改 SDD
#    - 旧的 specs/ 可以保留或删除（看是否仍有参考价值）

# 4. 更新顶层 docs/README.md
#    - 当前版本指向 v1.1
#    - v1.0 移到"历史版本"列表

# 5. （可选）在 v1.1/prd/codeask.md 的 §9 增加"与 v1.0 PRD 的差异"小节
```

新建 v2.0 流程同理，复制源是当前 latest 版本（不一定是 v1.0）。

## 10. 一些反例（**不要**这样做）

| 反例 | 为什么错 |
|---|---|
| `docs/v1.0/prd/2026-04-29-codeask.md` | 文件名带日期 + 版本号已在外层，重复 |
| `docs/v1.0/prd-v1.0/codeask.md` | 子目录里再嵌版本号 |
| `docs/v1.0/design-2026Q2/` | 子目录用日期分组 |
| 直接在 `v1.0/prd/codeask.md` 上做"翻天覆地的重写" | MAJOR 改动应该开 `v2.0/`，不是覆盖 v1.0 |
| 删掉 `v1.0/` 因为"已经过时了" | 历史不能删，永远保留 |
| `docs/PRD.md`（不在版本目录内） | 所有产品文档必须在某个 `vN.M/` 内 |
| `docs/v1.0/STRUCTURE.md`（重复约定） | 约定文件只在顶层 `docs/STRUCTURE.md`，版本目录内部不重复定义 |

## 11. 关于工具自动产物目录（已废弃）

**`docs/superpowers/` 等工具自动生成的子目录不再使用**，所有有保留价值的产物直接落到本文规定的版本树里：

| 工具产物类型 | 目标位置 | 命名规则 |
|---|---|---|
| 实现计划（writing-plans 等技能产出的 plan） | `vN.M/plans/<topic>.md` | 按主题，无日期前缀 |
| 设计草稿（brainstorming 等技能产出的 spec） | `vN.M/specs/<topic>.md` | 按主题，无日期前缀 |

无价值的、纯过程性的工具产物（运行日志、临时 dump 等）不进 `docs/`，由工具自行管理。

**反例**：
- `docs/superpowers/plans/2026-04-29-foundation.md`（中间层 + 日期前缀都不要）
- `docs/v1.0/superpowers/`（工具目录不进版本树）
- 任何为单个工具保留的顶层目录（`docs/<tool-name>/`）
