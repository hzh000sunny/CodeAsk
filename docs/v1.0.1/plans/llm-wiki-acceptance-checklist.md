## 1. 当前版本覆写规则

以下规则来自最近一轮联调结论，优先级高于旧文档中的软权限描述：

1. v1.0.1 第一版暂不启用 Wiki 写权限限制。
2. 当前所有用户默认都可以操作 Wiki，包括：
   - 上传 Markdown
   - 上传目录
   - 新建目录
   - 新建 Wiki
   - 编辑
   - 重命名
   - 删除
3. 管理员 / owner / member 的写权限区分，推迟到后续权限体系接入时再恢复。
4. 独立 Wiki 页面左侧目录树需要支持拖拽或显式拉伸，避免长目录名被硬截断后无法操作。
5. Wiki 导入交互从“preflight + apply”切换为“导入会话 + 文件队列”。
6. 导入抽屉必须展示完整文件列表；忽略文件位于列表最下方，默认折叠。
7. 上传目录时剥掉本地上传目录的最外层目录名，仅保留内部相对结构。
8. 冲突文件不阻断整批上传，必须支持 `覆盖 / 跳过 / 全部覆盖 / 全部跳过`。
9. 导入关闭抽屉或切页时，若未完成，必须弹确认框让用户选择继续后台上传或取消。

## 2. 验收状态定义

- `未开始`：未实现或未验证
- `进行中`：已有实现，但未完成联调或仍有缺陷
- `通过`：代码、前后端联调、真实浏览器交互均通过
- `失败`：当前行为与验收标准不符
- `暂缓`：不属于本轮 v1.0.1 收口范围，但必须保留后续接口或文档通道

## 3. 验收总原则

1. 每个功能项必须同时满足：
   - 文档契约存在
   - 前端行为可见
   - 后端接口可用
   - 至少有一类自动化测试覆盖
   - 关键主链路经过真实浏览器联调
2. 不允许只靠 mocked test 宣称通过。
3. 不允许“树能显示但点不开”、“按钮能看见但无法提交”、“接口存在但页面无入口”这类半成品状态通过验收。
4. 验收顺序按“先阻断主链路、再补管理能力、最后补边角能力”执行。

### 6.2 真实浏览器联调

必须逐项手工或脚本联调：

1. 打开 `#/wiki`
2. 验证默认首篇文档预览
3. 点击另一个特性，验证右侧预览切换
4. 点击具体文档，验证正文渲染
5. 进入编辑态，修改并刷新，验证草稿恢复
6. 发布后查看历史版本
7. 打开导入抽屉，选择单 Markdown，验证队列生成
8. 打开导入抽屉，选择目录，验证忽略文件折叠区
9. 验证逐文件进度、冲突处理、失败重试和后台继续上传
10. 验证导入完成后树刷新并自动打开目标一级目录下第一篇 Markdown
11. 验证删除、重命名、空态、新建、报告投影
12. 归档一个特性，验证它从默认特性列表消失，但仍能在 Wiki `历史特性` 下打开原文档

## 7. 本轮收口顺序

按阻断程度排序，必须按以下顺序验收：

1. `WIKI-019` ~ `WIKI-022`
   - 写入口全部恢复可见可用
2. `WIKI-023`
   - 每条目录右侧三点菜单位置和能力统一
3. `WIKI-033` ~ `WIKI-042`
   - 导入会话 / 文件队列 / 冲突处理 / 失败重试 / 后台继续上传链路打通
4. `WIKI-008` ~ `WIKI-009`
   - 左侧目录树拉伸和长名称可用性
5. `WIKI-029` ~ `WIKI-031`
   - 节点重命名 / 删除 / 确认链路
6. `WIKI-040` ~ `WIKI-041`
   - 报告投影和状态流转
7. `WIKI-010`、`WIKI-011`、`WIKI-016`
   - 搜索大小写不敏感、搜索分组和顶部动作收口
