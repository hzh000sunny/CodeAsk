# LLM Wiki 产品需求文档

> 本文档属于 v1.0.1 PRD，定义 CodeAsk LLM Wiki 专项的产品契约。
>
> v1.0.1 的设计和实现必须以本文为准。若 SDD、计划或早期讨论记录与本文冲突，以本文为准。

## 1. 版本定位

v1.0.1 是 CodeAsk 的 LLM Wiki 专项版本，目标是在 v1.0 已完成的“会话、特性、设置、Agent 调查、报告回流”主链路上，补齐独立、可治理、可被 Agent 稳定引用的 Wiki 知识库能力。

本版本不是在特性页里继续堆上传和预览能力，而是把 Wiki 建设为独立一级工作台：

```text
会话
特性
Wiki
设置
```

Wiki 是研发知识基础设施。它服务两个方向：

- 用户可以像维护普通 Markdown Wiki 一样维护特性知识、环境说明、Prompt 上下文和问题定位报告。
- Agent 可以在问题分析中透明地解析、检索、回源和引用 Wiki 内容，而不是只消费不可追溯的 snippet。

## 2. 用户与场景

主要用户：

| 用户 | 诉求 |
|---|---|
| 普通研发 / oncall | 查看特性知识、历史问题报告和排查手册，快速理解问题上下文 |
| 特性 owner | 维护本特性的知识库、报告验证状态、目录结构和引用资源 |
| 管理员 | 管理全局可见能力，恢复历史特性，处理长期治理和危险操作 |
| Agent | 根据用户问题、分析策略和特性上下文解析 Wiki 范围，检索并回源引用证据 |

核心场景：

1. 研发进入 Wiki，按特性浏览 `知识库` 和 `问题定位报告`。
2. 特性 owner 上传一个目录，保留 Markdown 相对路径和图片引用。
3. 用户编辑 Markdown，编辑中自动保存草稿，正式保存后生成版本并自动索引。
4. 会话生成的问题定位报告回流到对应特性的 Wiki 报告目录。
5. owner 验证报告后，该报告进入后续 Agent 的高优先级检索范围。
6. Agent 在运行事件中展示识别到的 Wiki 范围，并在最终回答中给出可点击 Wiki 证据路径。

## 3. 产品原则

### 3.1 独立一级模块

Wiki 必须是独立一级模块，不再嵌套在特性页面的狭小 tab 中。

特性页保留轻量入口：

- 展示当前特性的完整目录树。
- 渲染当前特性的 Markdown 预览。
- 展示当前特性的问题定位报告预览。
- 展示索引状态摘要。
- 对上传、编辑、移动、删除、版本历史等动作提供跳转入口。

复杂管理动作必须跳转到独立 Wiki 页面完成，不在特性页内部打开复杂表单。

特性页知识库预览的当前版本约束：

- 左侧默认只展开 `知识库` 根目录。
- `知识库` 下层目录默认收起，但允许用户逐层展开 / 收起。
- 右侧预览区只保留 Markdown 正文，不显示文档名头部，不显示“当前特性共有多少篇报告”这类与正文阅读无直接关系的信息。
- 预览区的目标是快速阅读当前文档，而不是复刻独立 Wiki 页的阅读态元信息结构。

### 3.2 一级目录由特性驱动

Wiki 根目录不允许用户自由创建一级目录。一级空间必须由特性列表创建，并带有 owner、描述、关联仓库和分析策略。

全局目录树固定分为：

```text
Wiki/
├── 当前特性/
└── 历史特性/
```

`历史特性` 默认折叠。删除特性不是物理删除 Wiki，而是把该特性和对应 Wiki 空间归档到 `历史特性`。

### 3.3 系统目录与自定义目录并存

每个特性空间默认创建两个系统目录：

```text
特性名/
├── 知识库/
└── 问题定位报告/
```

用户可以在特性空间下创建与这两个系统目录平级的自定义目录，例如 `客户环境`、`灰度规则`、`Prompt 上下文`。

系统目录语义：

| 目录 | 语义 |
|---|---|
| `知识库` | 当前特性的常青知识，默认参与 Agent 自动检索 |
| `问题定位报告` | 会话生成报告的 Wiki 投影，已验证报告默认高优先级参与检索 |

`问题定位报告` 下的 `草稿`、`已验证`、`未通过` 是虚拟状态分组，由报告生命周期驱动，不是普通目录。

### 3.4 Markdown-only

v1.0.1 的 Wiki 只支持 Markdown 作为可编辑知识格式。

明确支持：

- 新建 Markdown。
- 上传 Markdown。
- 上传包含 Markdown 和其引用资源的目录。
- Markdown 源码编辑。
- Markdown 渲染预览。
- Markdown 相对图片和相对文档链接解析。

明确不做：

- 富文本编辑器。
- 所见即所得编辑器。
- PDF / DOCX / TXT 作为原生 Wiki 内容。
- 多格式混合知识库。

如果后续需要支持 PDF / DOCX，应作为“导入并转换为 Markdown”的独立能力设计。

### 3.5 当前版本权限覆写

Wiki 默认全员可读。

为了先打通 v1.0.1 第一版的 Wiki 建设主链路，当前版本临时放开 Wiki 写能力：所有用户都可以执行创建、上传、编辑、重命名、删除等写操作。

| 操作 | 当前 v1.0.1 第一版 |
|---|---:|
| 查看目录树 | 可以 |
| 查看文档和报告 | 可以 |
| 搜索 Wiki | 可以 |
| 创建目录 / 文档 | 可以 |
| 上传 Markdown / 目录 | 可以 |
| 编辑 / 重命名 / 移动 | 可以 |
| 软删除 | 可以 |
| 验证 / 驳回报告 | 可以 |

后续正式权限体系接入后，再恢复为 `owner / admin 可写，普通用户只读`。因此权限通道必须保留：

- admin cookie 仍表示管理员身份。
- `subject_id == feature.owner_subject_id` 仍作为未来 owner 判定通道保留。
- 当前前后端实现不以该判定限制 Wiki 写操作，只保留接口和字段。
- 后续接入 AuthProvider 后替换身份来源，不改变 Wiki 权限接口。

## 4. 信息架构

### 4.1 阅读态

默认阅读态：

```text
左侧：目录树
右侧：Markdown 渲染预览
```

阅读态偏阅读，不设置常驻第三栏。详情、索引状态、引用关系、历史版本通过轻量入口打开。

右上角轻量操作：

```text
详情 / 复制链接 / 编辑 / 更多
```

其中 `更多` 负责承载次级动作，当前版本至少包含：

- 历史版本
- 导入 Wiki

当前 v1.0.1 第一版不在前端隐藏 Wiki 写操作入口。阅读态和目录树菜单对所有用户统一显示可用动作，后端保留未来权限收敛通道。

目录树每个节点可有三点菜单。当前版本三点菜单至少承载：

- 新建目录
- 新建 Wiki
- 导入 Wiki
- 重命名
- 删除

系统目录和报告投影节点继续按节点类型裁剪菜单能力。

### 4.2 编辑态

点击编辑后进入编辑态：

```text
左侧：Markdown 源码编辑区
右侧：实时 Markdown 渲染预览
```

编辑态默认收起目录树，只保留一个可展开目录按钮，让源码和预览有足够宽度。

阅读态下左侧目录树需要支持显式拉伸；过长目录名至少要支持悬停查看完整标题。

编辑过程中自动保存草稿。只有用户点击保存或发布，才表示编辑完成。

### 4.3 搜索

Wiki 独立页面的搜索默认覆盖全 Wiki 可读范围。

搜索结果需要分组展示，且匹配必须大小写不敏感：

| 分组 | 说明 |
|---|---|
| 当前特性 | 当前正在查看或当前会话关联特性的知识 |
| 问题定位报告 | 当前特性下的报告命中，优先展示已验证报告 |
| 其它当前特性 | 其它 active 特性的命中 |
| 历史特性 | 已归档特性的命中，需要显著提示历史性质 |

点击结果后打开目标 Markdown 或报告，并定位到 heading / chunk。历史特性结果打开后必须提示内容可能已经过期。

## 5. 文档生命周期

### 5.1 创建和上传

可写目录菜单提供：

- 新建 Markdown。
- 新建目录。
- 上传 Markdown。
- 上传目录。

上传目录必须保留相对路径。上传前执行导入预检查。

上传交互约束：

- 导入抽屉同时提供 `选择 Markdown` 与 `选择目录` 两个独立入口，不把两种选择方式混在一个控件里。
- 选中文件或目录后，页面立即显示完整文件队列，而不是单独的 preflight 结果页。
- 队列顶部显示目标目录、总进度、成功 / 冲突 / 失败 / 忽略 / 跳过 计数，以及当前正在处理的文件名。
- 文件列表默认显示全部文件，按用户选择顺序排列；每行显示文件名、相对路径、状态和进度条。
- 上传中的进度条不能只在请求结束后一次性跳到 100%；前端需要在单文件上传过程中实时刷新当前项进度，并同步更新顶部总进度与“当前处理文件”。
- 冲突项和失败项必须在队列行内直接展示原因，用户不需要额外打开二级详情才能知道为什么冲突或为什么失败。
- 过滤后不会上传的文件也要出现在列表中，但统一放在最底部的“已忽略文件”折叠区。
- 上传目录时，目标落点应直接使用用户选中的 Wiki 目录，剥掉本地上传目录的最外层目录名，只保留其内部相对结构。
- 导入完成后自动关闭抽屉，并默认打开目标一级目录下的第一篇 Markdown 预览。

冲突策略：

- 冲突不再阻断整个导入会话；冲突文件进入 `冲突待处理`，其它文件继续上传。
- 单个文件上传失败也不阻断后续队列；失败项保留在当前会话中，后续文件继续上传，用户稍后可单独重试或批量重试失败项。
- 冲突操作直接在当前文件行内展开，不额外跳转到独立确认页。
- 单文件至少提供：`覆盖`、`跳过`。
- 队列级至少提供：`全部覆盖`、`全部跳过`。
- 用户主动跳过的文件必须记录为 `已跳过`，不能混同于系统自动 `已忽略`。
- 关闭抽屉或切页时，若导入未结束，必须弹确认框，让用户选择继续后台上传或取消本次导入。
- 失败文件支持单个重试和“重试全部失败文件”。
- 默认并发上传数为 3。

相对引用策略：

- Markdown 引用的图片资源进入 Wiki 资源库。
- Markdown 相对链接到另一个 `.md` 时解析为 Wiki 内部链接。
- Markdown 相对链接点击后，必须直接打开对应 Wiki 节点，不要求用户手工回到目录树重新选择。
- 断链只警告，不阻断导入。
- 断链状态在导入结果和文档详情中展示。
- 对已经断链或运行时加载失败的图片，正文区要给出可见占位反馈，不能只剩浏览器默认破图状态。

导入状态语义：

- `待上传`
- `上传中`
- `已上传`
- `冲突待处理`
- `失败`
- `已忽略`
- `已跳过`

### 5.2 草稿和版本

编辑过程中自动保存草稿。

草稿语义：

- 草稿防止编辑丢失。
- 草稿不进入正式预览。
- 草稿不触发索引。
- 草稿不参与 Agent 检索。

正式保存语义：

- 每次保存生成一个正式版本。
- 版本记录保存人、保存时间、版本号、内容快照和可选变更摘要。
- 支持查看历史版本。
- 支持当前版本与历史版本对比。
- 支持回滚到历史版本。
- 回滚本身生成新的正式版本。
- 版本历史不进入目录树，只在文档详情或历史版本入口查看。

### 5.3 删除和恢复

Markdown 文档、普通目录、资源采用软删除。

软删除规则：

- 记录删除人、删除时间、删除前路径和节点类型。
- 软删除内容可恢复。
- v1.0.1 不做独立回收站页面。
- 软删除默认保留 30 天，超过后自动清理。
- 文档历史版本长期保留，不自动清理。
- 历史特性长期保留，不自动清理。
- 如果资源仍被任一历史版本引用，不能物理清理。

删除文档或目录后，对应正式内容必须自动从索引下架。

## 6. 报告回流

报告生命周期仍归 `reports` 管理。Wiki 只把报告投影到目录树中。

展示路径：

```text
特性名 / 问题定位报告 / 草稿 / 报告标题
特性名 / 问题定位报告 / 已验证 / 报告标题
特性名 / 问题定位报告 / 未通过 / 报告标题
```

状态规则：

- 会话生成报告后，绑定特性并显示在 `草稿`。
- 点击验证通过后进入 `已验证`。
- 点击验证不通过后进入 `未通过`。
- 已验证报告默认参与当前特性 Agent 检索。
- 草稿和未通过报告不参与默认检索。
- 报告不能通过拖拽移动改变状态，必须通过生命周期动作改变状态。

## 7. Agent 接入

Agent 引用 Wiki 不要求用户写特殊语法。系统应支持口语化描述。

示例：

```text
优先参考订单中心下面的 Prompt 上下文。
这个客户是 A 集群，先看订单中心里的客户环境说明。
可以参考历史特性里的旧版支付网关知识库，但要注意可能过期。
```

运行时需要解析为稳定范围：

```text
自然语言描述
→ 提取特性名、目录名、历史特性提示
→ 解析为候选 wiki node_id
→ 加入本轮检索范围
→ 在运行事件中展示解析结果
```

Agent 最终回答必须使用可点击 Wiki 路径作为证据引用，例如：

```text
Wiki：订单中心 / 知识库 / 错误码 / 订单提交失败.md#ERR_ORDER_CONTEXT_EMPTY
报告：订单中心 / 问题定位报告 / 已验证 / 2026-05-01 订单上下文为空
```

检索链路必须支持回源：

```text
search_wiki
→ 返回候选 node / chunk
→ read_wiki_node 回源读取原文
→ 最终回答引用 Wiki path + heading / chunk
```

当前版本落地口径：

- v1.0.1 当前已落地 Wiki 文档的 node 级与 heading 级可点击回链，确保会话回答可以直接跳转回独立 Wiki 页面并落到对应标题。
- v1.0.1 当前已要求 Agent 在 `scope_detection` 选中特性后，后续 Wiki 检索必须沿该特性集合执行；不允许知识检索阶段退回到“全特性无差别搜索”。
- v1.0.1 当前已支持跨多个选中特性聚合 Wiki 命中，范围事件和证据链接必须保留各自所属的 `feature_id`，避免跳回错误特性。
- 问题报告当前仍以 node 级跳转为主，后续如有稳定标题结构再补充更细粒度锚点。

## 8. 索引原则

任何影响正式内容、路径、报告状态或资源引用的操作完成后，都必须自动更新索引或从索引下架。

必须自动处理索引的操作：

| 操作 | 索引行为 |
|---|---|
| 新建 Markdown 并正式保存 | 建立索引 |
| 编辑后正式保存 | 更新索引 |
| 回滚历史版本 | 更新索引 |
| 上传 Markdown | 建立索引 |
| 上传目录成功 | 批量建立索引 |
| 删除文档 | 下架索引 |
| 移动 / 重命名文档 | 更新路径和内容索引 |
| 移动 / 重命名目录 | 更新目录下所有文档路径索引 |
| 报告验证通过 | 进入报告索引 |
| 报告撤销验证 / 标记未通过 / 删除 | 下架报告索引 |
| 断链状态变化 | 更新文档元信息索引 |

手动重新索引只作为修复入口，不是正常工作流。

## 9. 非目标

v1.0.1 不做以下事情：

- 完整企业登录和权限系统。
- Wiki 根目录自由创建一级目录。
- 富文本编辑器。
- PDF / DOCX / TXT 原生 Wiki 内容。
- Git-backed Wiki。
- 独立回收站页面。
- 历史特性永久删除。
- 复杂审批流。
- 向量库优先方案。
- 大范围重构会话、特性、设置主链路。

## 10. 验收标准

产品验收至少包括：

- 导航中出现独立 `Wiki` 一级入口。
- Wiki 页面可查看 `当前特性` 和默认折叠的 `历史特性`。
- 每个特性空间自动具备 `知识库` 和 `问题定位报告`。
- 特性页只保留当前特性 Wiki 树和预览，复杂操作跳转到 Wiki。
- 特性页 `知识库` 预览树默认只展开 `知识库` 根目录；右侧预览只显示正文，不显示文档名头部和特性级报告计数。
- Markdown 文档可以新建、编辑、预览、正式保存、查看版本、回滚。
- 编辑中自动保存草稿，刷新后能继续编辑。
- 目录上传保留相对路径，并剥掉本地上传目录的最外层目录名。
- 导入抽屉显示完整文件队列、逐文件进度和忽略文件折叠区。
- 冲突文件不阻断整批导入，可在当前行内执行覆盖 / 跳过 / 全部覆盖 / 全部跳过。
- 相对图片可渲染，断链可见但不阻断。
- 软删除可恢复，30 天后清理普通软删除内容。
- 报告在 Wiki 中按虚拟状态分组展示，验证状态驱动检索资格。
- 当前版本全员可读可写，后续再恢复 owner / admin 写权限收敛。
- 所有正式变更自动触发索引更新或下架。
- Agent 运行事件展示 Wiki 范围解析。
- Agent 最终回答中的 Wiki 证据是可点击路径。
