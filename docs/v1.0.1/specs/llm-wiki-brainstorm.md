# LLM Wiki 头脑风暴记录

> 本文记录 v1.0.1 LLM Wiki 专项当前已经讨论并确认的方向，避免上下文丢失。
>
> 本文是讨论记录和设计草稿，不是最终 PRD / SDD。后续定稿后应拆分为 `prd/llm-wiki.md`、`design/llm-wiki-workbench.md` 和 `plans/llm-wiki-workbench.md`。

## 0. 阶段性设计摘要

当前已经确认的 v1.0.1 LLM Wiki 设计骨架如下。

产品定位：

- v1.0.1 专门用于补齐 LLM Wiki 能力。
- Wiki 是独立一级模块，导航中新增 `Wiki`。
- 当前特性页保留当前特性的目录树和 Markdown 预览，但上传、编辑、移动、删除、版本历史、导入任务等复杂管理动作跳转到独立 Wiki 页面完成。
- Wiki 当前只支持 Markdown 作为可编辑知识格式，不做富文本、不做 PDF / DOCX / TXT 原生 Wiki 内容。

目录和空间：

- Wiki 根目录不允许用户自由创建一级目录。
- 一级目录必须由特性列表驱动。
- 全局目录树固定分为 `当前特性` 和 `历史特性`，其中 `历史特性` 默认折叠。
- 每个特性目录默认包含系统目录 `知识库` 和 `问题定位报告`。
- 用户可以在特性目录下创建与 `知识库`、`问题定位报告` 平级的自定义目录。
- 删除特性是归档，不清理 Wiki；对应 Wiki 移入 `历史特性`。

检索和报告：

- `知识库/` 默认参与当前特性的常青知识检索。
- `问题定位报告/已验证` 默认参与当前特性的高优先级报告检索。
- `问题定位报告/草稿`、`问题定位报告/未通过` 不参与默认检索。
- 平级自建目录默认不参与 Agent 自动检索，但可以被分析策略或用户问题口语化引用后加入范围。
- `问题定位报告` 下的 `草稿`、`已验证`、`未通过` 是虚拟目录，由报告状态驱动，不能通过拖拽绕过验证流程。

权限：

- Wiki 默认全员可读。
- 只有特性 owner 和管理员可以创建、上传、编辑、移动、删除、恢复、验证报告等写操作。
- 当前无完整登录鉴权，owner 判定先通过软 `subject_id == feature.owner_subject_id` 预留通道；后续接入真实 AuthProvider 后替换身份来源。

阅读和编辑布局：

- 阅读态：左侧目录树，右侧 Markdown 渲染预览。
- 阅读态偏阅读，不做常驻第三栏；详情、索引状态、引用关系、历史版本入口通过右侧抽屉或轻量浮动操作打开。
- 阅读态右上侧提供轻量浮动操作：`详情 / 复制链接 / 编辑 / 更多`，按权限裁剪。
- 目录树每个节点有三点下拉菜单。
- 编辑态：左侧 Markdown 源码编辑区，右侧实时 Markdown 渲染预览。
- 编辑态默认收起目录树，只保留可展开目录按钮。

文档生命周期：

- 目录树可写目录菜单提供 `新建 Markdown`、`新建目录`、`上传 Markdown`、`上传目录`。
- 目录上传前做导入预检查。
- 同名文件或目录冲突阻断导入，列出冲突路径，用户必须手动处理后重试。
- Markdown 相对图片和内部链接需要解析；断链只警告，不阻断导入。
- 编辑过程自动保存草稿防丢失；草稿不进入正式预览、不触发索引、不参与 Agent 检索。
- 用户点击保存后生成正式版本，并触发索引。
- 版本历史是 v1.0.1 必备能力：每次正式保存生成版本，支持查看、对比、回滚；回滚本身生成新正式版本。
- 文档、普通目录、资源删除采用软删除，保留 30 天后自动清理。
- 文档历史版本和历史特性长期保留，不自动清理。

索引和 Agent：

- 任何影响正式内容、路径、报告状态或资源引用的操作完成后，都自动更新索引或从索引下架。
- 手动重新索引只作为修复入口，不是正常工作流。
- Agent 引用 Wiki 目录采用口语化描述，由 AI / 后端目录解析服务识别，不要求用户写 `@wiki:`。
- Wiki 目录解析结果必须在会话运行事件中展示。
- Agent 最终回答中的 Wiki 证据必须使用可点击路径，包含特性、目录、文档和 heading。

搜索：

- 独立 Wiki 页搜索默认覆盖全 Wiki 可读范围。
- 搜索结果按当前特性、问题定位报告、其它当前特性、历史特性分组展示。
- 点击结果打开目标 Markdown 或报告，滚动到命中 heading / chunk 并高亮。
- 历史特性结果需要明显提示其历史性质。

## 1. 版本判断

LLM Wiki 建议放入 `v1.0.1`，作为专门版本开发。

理由：

- 当前 v1.0 主链路已经形成：会话、特性、设置、Agent、报告回流。
- 完整 LLM Wiki 是 v1.0 已明确后置的核心缺口，而不是一个完全独立的新产品方向。
- v1.0.1 不改变主链路，主要补齐知识库管理能力，使现有 Agent 和报告飞轮可持续运转。

风险：

- 从严格语义化版本看，完整 Wiki 工作台也可以视为 `v1.1` 级别能力增强。
- 若后续讨论中引入企业级权限、跨空间协作、复杂审核流等大范围产品变化，应重新评估是否改为 `v1.1`。

当前结论：

> 暂按 v1.0.1 推进，版本主题限定为 LLM Wiki 专项。

## 2. 当前 Wiki 能力缺口

当前 v1.0 的 Wiki 更接近“文档上传 + 切块检索 + 报告回流”的 MVP 骨架，不是完整 LLM Wiki。

主要欠缺：

| 类别 | 缺口 |
|---|---|
| 页面形态 | Wiki 嵌在特性详情 tab 中，空间狭窄，不适合完整知识管理 |
| 目录模型 | 前端看似支持目录上传，但后端只保存 basename，没有真实目录树 |
| 文件冲突 | 同名文件可能冲突，无法区分不同目录下的同名日志或文档 |
| 相对资源 | Markdown 图片、链接只记录引用，不校验、不服务、不修复 |
| 内容预览 | 当前只预览文档元信息，不渲染 Wiki 文档正文 |
| 编辑维护 | 缺少编辑、重命名、移动、替换、版本、diff、回滚、重新索引 |
| 检索质量 | FTS / n-gram 有基础能力，但精确信号召回、metadata 加权、rerank 不足 |
| Agent 回源 | 设计上要求 search 后 read 原文，当前主链路更多依赖 snippet evidence |
| 知识治理 | 缺少断链检测、过期检测、重复检测、命中统计、质量提示 |
| 报告飞轮 | 报告状态和适用条件仍需加强，历史报告默认检索策略未细化 |

## 3. 已确认产品原则

### 3.1 Wiki 是独立一级模块

Wiki 不应继续作为特性详情中的狭小面板存在。应新增独立一级导航：

```text
会话
特性
Wiki
设置
```

特性页仍可保留轻量入口，但仅承担：

- 当前特性的 Wiki 路径展示。
- 当前特性的完整目录树浏览。
- Markdown 渲染预览。
- 问题定位报告预览。
- 索引状态摘要。
- 跳转到完整 Wiki 页面进行上传、编辑、移动、删除、版本历史等管理操作。

特性页不承载完整 Wiki 管理。上传、编辑、移动、删除、版本历史、导入任务等写操作应跳转到独立 Wiki 页面完成。

特性页中的目录树只显示当前特性的子目录：

```text
知识库
问题定位报告
客户环境
Prompt 上下文
```

不显示全局 `当前特性 / 历史特性` 分组，也不显示其它特性。完整 Wiki 页面才展示全局目录树。

特性页可以展示上传、编辑等操作入口，但点击后跳转到独立 Wiki 页面执行，不在特性页内打开复杂管理表单。

示例：

```text
/wiki?node=123&action=edit
/wiki?parent=456&action=upload-markdown
/wiki?parent=456&action=upload-directory
/wiki?node=789&action=history
```

这样特性页保留入口完整性，复杂操作仍集中在独立 Wiki 工作台。

### 3.2 Wiki 一级目录必须由特性驱动

Wiki 根目录不允许用户自由创建一级目录。

一级目录必须来自特性列表，具有强关联性和 owner：

```text
Wiki/
├── 当前特性/
│   ├── 订单中心/
│   ├── 风控策略/
│   └── 数据库集群/
└── 历史特性/
    ├── 旧版支付网关/
    └── 已下线订单服务/
```

目录树采用固定分组展示：

- `当前特性`
- `历史特性`

`历史特性` 默认折叠。这样用户能明确知道历史知识仍被保留，但不会干扰日常 Wiki 阅读和维护。

如果用户需要“全局规范”“运维手册”“Prompt 上下文”等能力，应在特性列表中创建对应特性，例如：

- 全局研发规范
- 数据库平台
- 运维体系
- Prompt 上下文

这样一级空间始终有 owner、描述、关联仓库和分析策略，不会变成无治理的公共目录。

### 3.3 每个特性目录有系统目录

每个特性目录下默认创建两个最大目录：

```text
订单中心/
├── 知识库/
└── 问题定位报告/
```

用户可以在特性目录内部继续创建其它目录，且自建目录可以与 `知识库`、`问题定位报告` 平级：

```text
订单中心/
├── 知识库/
│   ├── 架构设计/
│   ├── 错误码/
│   ├── 运维手册/
│   └── images/
├── 问题定位报告/
│   ├── 草稿/
│   ├── 已验证/
│   └── 未通过/
├── 客户环境/
├── 灰度规则/
└── Prompt 上下文/
```

`知识库` 和 `问题定位报告` 是系统目录，有特殊语义：

- `知识库`：默认参与当前特性的常青知识检索。
- `问题定位报告`：承载会话生成的报告，验证后高优先级参与检索。

其它子目录是用户目录，可以被 prompt / 分析策略显式引用。

`问题定位报告` 下的状态目录是虚拟目录，由报告状态驱动，不是普通用户目录：

```text
问题定位报告/
├── 草稿
├── 已验证
└── 未通过
```

报告不能通过拖拽或移动进入 `已验证`。必须执行“验证通过”动作，报告状态变更后才显示在 `已验证` 分组中。这样避免用户绕过报告验证流程污染检索。

默认检索范围：

| 目录 | 默认是否参与 Agent 自动检索 | 说明 |
|---|---:|---|
| `知识库/` | 是 | 当前特性的常青知识 |
| `问题定位报告/已验证` | 是 | 当前特性的高优先级历史案例 |
| `问题定位报告/草稿` | 否 | 未验证，不进入默认检索 |
| `问题定位报告/未通过` | 否 | 已确认不通过，不进入默认检索 |
| 平级自建目录 | 否 | 可被分析策略或用户问题显式路径引用后参与 |

这样可以避免 `客户环境/`、`Prompt 上下文/`、`实验规则/` 等特殊目录意外污染所有普通问答。

### 3.4 删除特性必须归档 Wiki，不清理知识

删除特性不应物理删除对应 Wiki。

应将特性和 Wiki 空间转入历史特性：

```text
/当前特性/订单中心
→
/历史特性/订单中心
```

产品语义更接近“归档特性”，不是彻底删除。

建议确认弹窗文案：

```text
确认归档这个特性？
归档后该特性不会出现在默认特性列表和 Agent 默认定界中，
对应 Wiki 会移动到历史特性目录，文档和报告不会被删除。
```

历史特性的默认行为：

- 所有人仍可查看。
- 默认不参与普通会话的 Agent 自动检索。
- 用户显式指定历史特性路径时可以检索，但回答应提示其历史性质。
- 管理员可以恢复历史特性。
- 永久删除应作为管理员危险操作单独设计。

历史特性展示和恢复规则：

- 历史特性在目录树 `历史特性` 分组下长期展示。
- 历史特性默认只读为主。
- owner / admin 可以编辑历史 Wiki 内容，但仍不参与默认 Agent 自动检索。
- 管理员可以执行“恢复特性”，恢复后移动回 `当前特性` 分组。
- 恢复时如果当前特性中存在同名特性，阻断并提示冲突，需要先重命名。
- 历史特性不能被普通删除；管理员永久删除作为危险操作另行设计。

### 3.5 Wiki 默认全员可读，owner / admin 可写

目标权限模型：

| 操作 | 普通用户 | 特性 owner | 管理员 |
|---|---:|---:|---:|
| 查看目录树 | 可以 | 可以 | 可以 |
| 查看文档内容 | 可以 | 可以 | 可以 |
| 搜索 Wiki | 可以 | 可以 | 可以 |
| 查看问题定位报告 | 可以 | 可以 | 可以 |
| 创建目录 | 不可以 | 可以 | 可以 |
| 上传文档 / 资源 | 不可以 | 可以 | 可以 |
| 编辑文档 | 不可以 | 可以 | 可以 |
| 重命名 / 移动 | 不可以 | 可以 | 可以 |
| 删除 / 归档文档 | 不可以 | 可以 | 可以 |
| 重新索引 | 不可以 | 可以 | 可以 |
| 验证 / 驳回报告 | 建议不可以 | 可以 | 可以 |
| 删除报告 | 不可以 | 可以 | 可以 |
| 恢复历史特性 | 不可以 | 建议不可以 | 可以 |
| 永久删除历史 Wiki | 不可以 | 不可以 | 可以 |

当前没有完整登录鉴权，需保留权限通道：

- 管理员：已有 admin cookie。
- owner：当前请求 `subject_id == feature.owner_subject_id`，属于软身份。
- 普通用户：其它 subject_id。

当前实现不应把 owner 判定宣传为强安全边界。后续接入 AuthProvider 后替换身份来源。

建议 API 返回权限字段：

```json
{
  "permissions": {
    "can_read": true,
    "can_write": false,
    "can_delete": false,
    "can_admin": false,
    "reason": "viewer"
  }
}
```

## 4. 初步信息架构

独立 Wiki 页面应围绕 Markdown 阅读和编辑两种状态设计。当前已明确：v1.0.1 的 Wiki 只支持 Markdown 作为可编辑知识格式，不做富文本、不做 DOCX / PDF 原生编辑。

### 4.1 阅读态

阅读态是默认状态：

```text
左侧：目录树
右侧：Markdown 渲染预览
```

阅读态用于日常浏览、搜索后阅读、从特性页跳转查看知识。正文区域应优先保证阅读宽度。文档元信息、索引状态、引用关系可以通过右侧抽屉、顶部信息区或详情按钮按需打开，不应长期占用正文布局。

阅读态不设置常驻第三栏。文档详情采用右侧抽屉或顶部“详情”按钮触发，包含：

- 文档路径。
- 创建人 / 更新人。
- 更新时间。
- 索引状态。
- 引用关系。
- 断链提示。
- 历史版本入口。

阅读态工具栏应偏阅读，不偏后台管理。正文区域不堆大量操作按钮。管理动作采用轻量浮动入口：

```text
右上轻便悬浮操作：
详情 / 复制链接 / 编辑 / 更多
```

普通用户只看到阅读相关动作：

```text
详情 / 复制链接
```

特性 owner / 管理员额外看到：

```text
编辑 / 移动 / 删除 / 重新索引
```

目录树的每个 Wiki 节点可以有三点下拉菜单。节点菜单根据权限展示：

| 节点类型 | 普通用户 | owner / admin |
|---|---|---|
| 目录 | 打开、复制路径 | 新建子目录、上传、重命名、移动、删除、重新索引 |
| Markdown 文档 | 打开、复制链接 | 编辑、重命名、移动、删除、重新索引 |
| 资源 | 打开、复制链接 | 重命名、移动、删除 |
| 报告引用 | 打开、复制链接 | 验证、标记未通过、撤销验证、删除引用 |

目录树任意可写目录的三点菜单提供 Markdown 创建入口：

```text
新建 Markdown
新建目录
上传 Markdown
上传目录
```

规则：

- `新建 Markdown`：创建空白 `.md` 文档，立即进入编辑态。
- `新建目录`：在当前目录下创建子目录。
- `上传 Markdown`：上传单个 `.md` 文件。
- `上传目录`：保留相对路径，导入目录内的 `.md` 和被 Markdown 引用的资源。
- PDF / DOCX / TXT 不作为 v1.0.1 Wiki 原生内容上传。

目录树移动规则：

- owner / admin 可以拖拽移动普通目录和 Markdown 文档。
- 普通用户不能移动节点。
- 系统目录 `知识库`、`问题定位报告` 不能移动、删除或改名。
- `问题定位报告` 下的状态虚拟目录不能拖拽。
- 报告不能通过拖拽改变状态。
- 移动后自动更新路径和索引。
- 如果移动造成路径冲突，阻断并提示冲突路径。

删除规则：

- Markdown 文档、普通目录、资源采用软删除。
- 删除动作记录历史，包含删除人、删除时间、删除前路径和节点类型。
- 软删除内容可恢复。
- v1.0.1 不做独立回收站页面。
- 恢复入口可以放在目录详情、历史记录或文档详情中。
- 软删除默认保留 30 天，超过后自动清除。
- 删除 Markdown 文档后自动从索引下架。
- 删除目录后目录下正式文档全部从索引下架。
- 删除资源时，如果仍被 Markdown 引用，需要提示引用关系；确认删除后相关文档标记断链。
- 系统目录 `知识库`、`问题定位报告` 不能删除。
- 历史版本不能被普通删除动作单独删除；管理员级永久清理另行设计。
- 文档历史版本长期保留，不自动清除。
- 历史特性长期保留，不自动清除。
- 资源文件如果仍被任一历史版本引用，不能被物理清理。

### 4.2 编辑态

点击编辑后进入源码编辑 + 实时预览：

```text
左侧：Markdown 源码编辑区
右侧：实时 Markdown 渲染预览
```

编辑态下目录树默认收起，只保留一个可展开目录按钮。这样源码和预览都有足够宽度；用户需要切换文档时，可以点击目录按钮临时展开目录树。

编辑态应显示当前文档路径，例如：

```text
订单中心 / 知识库 / 错误码 / 订单提交失败排查手册.md
```

### 4.3 格式边界

v1.0.1 的 Wiki 格式边界：

- 新建文档只创建 Markdown。
- 在线编辑只编辑 Markdown 源码。
- 预览只渲染 Markdown。
- 目录上传只接受 Markdown 文件和 Markdown 引用的资源。
- PDF / DOCX / TXT 不作为原生 Wiki 格式进入本阶段。
- 若后续需要支持 PDF / DOCX，应作为“导入并转换成 Markdown”能力单独设计。
- 不做富文本编辑器，不做 WYSIWYG，不做多格式混合知识库。

### 4.4 编辑保存语义

Markdown 编辑过程中自动保存草稿，避免浏览器刷新、网络中断或误关闭导致内容丢失。

草稿不代表正式内容：

- 自动保存草稿只保存编辑中的临时版本。
- 草稿不进入正式 Markdown 预览。
- 草稿不触发重新索引。
- 草稿不参与 Agent 检索。
- 只有用户点击“保存”或“发布”按钮，才表示编辑完成。
- 保存 / 发布后生成正式版本，并触发重新索引。
- 若用户离开编辑态时存在未发布草稿，UI 应提示继续编辑、丢弃草稿或保存为正式版本。

版本历史是 Wiki v1.0.1 必备能力，不后置：

- 每次正式保存生成一个版本。
- 草稿不算正式版本。
- 版本记录保存人、保存时间、版本号和内容快照。
- 版本可以附可选变更摘要。
- 支持查看历史版本。
- 支持当前版本与历史版本对比。
- 支持回滚到指定历史版本。
- 回滚本身也应生成一个新的正式版本，避免破坏历史链。

### 4.5 候选页面能力

候选页面能力：

- 搜索全 Wiki 可读范围。
- 浏览当前特性和历史特性。
- 展开目录树。
- 创建子目录。
- 上传文件或目录。
- 查看 Markdown 渲染结果。
- 查看原文。
- 编辑 Markdown。
- 重命名 / 移动 / 删除。
- 查看相对引用和断链。
- 查看索引状态。
- 手动重新索引。
- 从特性页或会话报告跳转到对应 Wiki 路径。

### 4.6 搜索结果

Wiki 独立页面的搜索默认覆盖全 Wiki 可读范围，而不是只搜索当前目录或当前特性。

搜索结果应分组展示，避免当前知识、其它特性知识和历史知识混在一起：

```text
搜索：ERR_ORDER_CONTEXT_EMPTY

当前特性
- 订单中心 / 知识库 / 错误码 / 订单提交失败.md
- 订单中心 / Prompt 上下文 / 用户上下文.md

问题定位报告
- 订单中心 / 问题定位报告 / 已验证 / 2026-05-01 订单上下文为空

其它当前特性
- 风控策略 / 知识库 / 上下文校验.md

历史特性
- 历史特性 / 旧版支付网关 / 知识库 / 上下文透传.md
```

历史特性结果应带明显提示，避免用户误以为它仍适用于当前系统。

搜索结果点击行为：

- 点击文档结果：打开对应 Markdown 文档，并滚动到匹配 heading / chunk。
- 匹配片段应高亮。
- 点击报告结果：打开报告预览，并定位到命中段落。
- 如果结果来自历史特性，打开后在页面顶部显示“历史特性内容”提示。
- 搜索结果列表应可保留为侧栏或抽屉，方便用户继续切换结果。

## 5. 初步数据模型方向

当前 `documents.feature_id` 的扁平模型不足以表达独立 Wiki 空间。后续需要引入节点模型。

候选模型：

```text
wiki_spaces
  id
  feature_id
  scope: current | history
  display_name
  slug
  archived_at
  archived_by
```

```text
wiki_nodes
  id
  space_id
  parent_id
  type: folder | document | asset | report_ref
  name
  path
  system_role
  sort_order
```

```text
wiki_documents
  node_id
  raw_file_path
  title
  content_text
  render_cache
  summary
  version
```

```text
wiki_assets
  node_id
  raw_file_path
  mime_type
  size
```

```text
wiki_report_refs
  node_id
  report_id
  status
```

路径设计原则：

- 内部使用稳定 `node_id`。
- UI 展示中文路径。
- prompt / 分析策略允许引用中文路径。
- 保存策略时尽量把路径解析成 `node_id`。
- 目录重命名后，绑定 `node_id` 的策略不应失效。

## 6. 上传和导入方向

目录上传不应继续作为多个同步文件上传请求处理。

建议引入导入任务：

```text
POST /api/wiki/imports
→ 创建导入任务
→ 保存原始目录结构
→ 处理同名和路径冲突
→ 保存 Markdown / 文本 / PDF / DOCX / 图片资源
→ 解析 Markdown 相对引用
→ 检查断链
→ 切 chunk
→ 建索引
→ 返回导入结果
```

导入结果应包含：

- 成功文件数量。
- 跳过文件数量。
- 同名冲突列表。
- 解析失败列表。
- 断链列表。
- 图片资源缺失列表。
- 索引状态。

冲突策略：

- 上传目录前必须做导入预检查。
- 如果目标路径下存在同名文件或目录冲突，导入失败。
- 系统列出全部冲突路径。
- 不自动覆盖。
- 不自动重命名。
- 不默认跳过。
- 用户必须手动处理冲突后重新导入。

这样可以避免批量导入误覆盖已有知识，保证 Wiki 内容变更是显式、可理解的。

相对资源和断链策略：

- 上传目录时允许 Markdown 引用同目录或子目录下的图片资源。
- 资源文件只作为 Markdown 引用资源进入库，不作为可编辑知识文档。
- Markdown 中的相对图片链接在预览时应正常渲染。
- Markdown 链接到另一个 `.md` 时，应解析为 Wiki 内部链接。
- 引用目标不存在时记录断链。
- 断链不阻断导入，只在导入预检查和文档详情中提示。
- 文件 / 目录同名冲突才阻断导入。

## 7. Agent 接入方向

Agent 检索 Wiki 时不应只按 feature 取 snippet。

### 7.1 自动索引原则

任何影响正式 Wiki 内容、路径、报告状态或资源引用的操作完成后，都应自动更新索引。索引不是用户额外需要记住的步骤。

必须自动处理索引的操作：

| 操作 | 索引行为 |
|---|---|
| 新建 Markdown 并正式保存 | 建立该文档索引 |
| 编辑后保存正式版本 | 更新该文档索引 |
| 回滚历史版本 | 更新该文档索引 |
| 上传 Markdown | 建立该文档索引 |
| 上传目录成功 | 批量建立新增文档索引 |
| 删除文档 | 从索引下架 |
| 移动 / 重命名文档 | 更新路径、标题和内容索引 |
| 移动 / 重命名目录 | 更新目录下所有文档路径索引 |
| 报告验证通过 | 进入报告索引 |
| 报告撤销验证 / 标记未通过 / 删除 | 从报告索引下架 |
| 断链状态变化 | 更新文档元信息索引或详情状态 |

手动“重新索引”只作为修复入口，用于索引损坏、导入异常或管理员排查，不应成为正常工作流的一部分。

后续工具方向：

```text
resolve_wiki_path(path)
list_wiki_children(node_id)
search_wiki(query, paths?, feature_ids?, node_types?)
read_wiki_node(node_id)
search_reports(query, paths?, status?)
read_report(report_id)
```

检索范围应支持：

| 范围 | 说明 |
|---|---|
| 当前特性默认范围 | 当前特性的 `知识库` 和已验证报告 |
| 分析策略指定范围 | 策略中显式引用的特性目录或子目录 |
| 用户会话临时范围 | 用户问题中显式引用的目录，或会话上传附件 |
| 历史特性范围 | 默认不参与，显式引用时参与，并提示历史性质 |

Wiki 目录引用应优先支持口语化表达，不要求用户或分析策略写特殊语法。

示例：

```text
分析这个问题时优先参考订单中心下面的 Prompt 上下文目录。
这个客户是 A 集群，先看订单中心里的客户环境说明。
可以参考历史特性里的旧版支付网关知识库，但要注意它可能已经过期。
```

运行时应由 AI / 后端目录解析服务识别目录意图：

```text
自然语言描述
→ 提取可能的特性名、目录名、历史特性提示
→ resolve_wiki_path / search_wiki_nodes
→ 得到候选 node_id
→ 在运行事件中展示解析结果
```

如果解析到多个候选目录，应让 Agent 根据上下文选择；不确定时可以向用户确认。

如果解析失败，应在运行事件中提示：

```text
未找到“订单中心 / Prompt 上下文”对应 Wiki 目录。
```

后续可选支持显式语法作为高级用法，但 v1.0.1 不要求用户写 `@wiki:` 这类标记。

目录解析结果必须在会话运行事件中展示，避免 Wiki 参考范围不透明。

示例事件：

```text
Wiki 目录解析

识别到：
- 订单中心 / Prompt 上下文
- 订单中心 / 客户环境

检索范围：
- 订单中心 / 知识库（默认）
- 订单中心 / 问题定位报告 / 已验证（默认）
- 订单中心 / Prompt 上下文（策略指定）
- 订单中心 / 客户环境（用户问题指定）
```

如果引用历史特性，应显式提示：

```text
已加入历史特性 / 旧版支付网关 / 知识库。该目录默认不参与当前问题检索，当前因用户显式提及而加入，内容可能过期。
```

可信引用链要求：

```text
search_wiki
→ 返回候选 node / chunk
→ read_wiki_node 回源读取原文
→ 最终回答引用 Wiki path + heading / line / chunk
```

最终回答中的 Wiki 证据引用应使用可点击路径，而不是只展示文档标题。

示例：

```text
证据：
- Wiki：订单中心 / 知识库 / 错误码 / 订单提交失败.md#ERR_ORDER_CONTEXT_EMPTY
- 报告：订单中心 / 问题定位报告 / 已验证 / 2026-05-01 订单上下文为空
- 代码：platform-api@abc123 src/order/service.py:88-104
```

点击 Wiki 引用应跳转到对应文档和 heading；点击报告引用应跳转到 Wiki 中的问题定位报告位置。

## 8. 已确认不能做的事情

当前已明确：

- 不允许在 Wiki 根目录自由创建一级目录。
- 不允许删除特性时清理 Wiki 内容。
- 不应把完整 Wiki 管理继续塞进特性页 tab。
- 不应在没有真实鉴权前假装 owner/admin 是强权限。
- 不应优先用向量库掩盖目录、预览、回源和治理缺失。

## 9. 待继续讨论

下一步建议逐项讨论：

1. Wiki 独立页面具体布局：三栏工作台，还是传统左树右文档阅读。
2. 特性页中的轻量 Wiki 概览保留到什么程度。
3. 历史特性目录的 UI 展示方式：单独分区、折叠分组，还是搜索中可见。
4. 文档编辑能力一期边界：只编辑 Markdown，还是支持富文本 / 所见即所得。
5. 目录上传冲突策略：覆盖、保留两份、提示用户选择，还是自动重命名。
6. 报告目录是否按状态物理分目录，还是只是虚拟过滤。
7. owner/admin 写权限在当前软身份阶段是否只做 UI 隐藏，还是 API 也做软校验。
8. prompt / 分析策略引用 Wiki 目录的语法。
9. 重新索引任务的粒度：单文件、目录、特性、全库。
10. 是否需要导入外部 Git 仓库中的 docs 目录作为 Wiki 源。
