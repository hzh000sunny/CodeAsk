# 前端研发工作台设计

> 本文档属于 v1.0 SDD，描述前端工作台的页面结构与交互。
>
> 产品契约见同版本 `prd/codeask.md`。当 SDD 与 PRD 冲突时，以 PRD 为准。

## 1. 目标

前端不是单纯聊天框，而是研发调查工作台。用户需要看到系统正在查什么、找到什么证据、当前结论是否受版本限制。

落地 PRD 的几条关键 UI 承诺分为当前阶段与后续指标阶段：

- **结论先 + 证据折叠**（PRD §8.4）：oncall 凌晨 2 点要能 5 秒看懂结论。
- **A2 / A3 透明**（PRD §7.1）：定界结果和"知识库够不够"判断必须可见，并提供兜底按钮。
- **deflection rate 反馈**（PRD §5.1）：当前 workbench 已接入 `/api/feedback` 和 `/api/events`，显式反馈与关键前端事件可进入 raw metrics 表。
- **Maintainer dashboard**（PRD §4.2）：Dashboard 仍是后续前端增强；它依赖已落地的 feedback / frontend_events / audit_log / agent_traces 数据面，不阻塞当前 workbench handoff。
- **不假装自信**（PRD §6.1）：建议操作要明确"是建议、决策权在人"。

## 2. 页面结构

> 2026-05-01 修订：全局配置不再对所有自报身份可见。普通用户仍可匿名使用，但全局配置、全局仓库和全局 LLM 配置只对管理员可见。详细契约见 `../specs/frontend-workbench-admin-rbac.md`。

一期工作台只暴露三个一级入口，避免把会话、知识库和系统配置混到同一个页面：

| 一级入口 | 页面内容 |
|---|---|
| 会话 | 会话列表、聊天流、调查进度、生成报告入口（§3-§7） |
| 特性 | 特性列表、特性设置、知识库、问题报告、关联仓库、特性分析策略 |
| 设置 | 普通用户：用户配置、个人 LLM 配置；管理员：全局配置、全局 LLM 配置、全局仓库池和全局分析策略，不展示个人用户配置 |

Maintainer Dashboard、报告详情页和更完整的 Wiki 编辑体验仍保留为后续增强；当前阶段把报告列表放在特性详情的问题报告 tab 中，把全局仓库注册放在管理员设置中。

特性列表顶部为搜索框 + 新建按钮。列表项展示特性名和 slug，右侧提供删除图标；点击删除必须先弹出确认框，确认后调用 `DELETE /api/features/{feature_id}`，成功后从当前列表移除并刷新特性查询。特性详情页只承载该特性的设置、知识库、问题报告、关联仓库和特性分析策略，不把删除动作藏到详情页内部。

全局配置中的仓库管理必须支持添加、编辑、同步 / 重试同步和删除。`刷新` 这个文案不再使用：对 `ready` 仓库使用 `同步`，表示同步 CodeAsk 后端用于代码检索和 Agent 调查的仓库缓存；对 `failed` 仓库使用 `重试同步`，表示重新投递 clone / fetch 任务。编辑仓库名称不触发同步；编辑 Git URL 或本地路径会让仓库重新进入同步流程。所有失败都必须有可见消息提示。

分析策略是 Prompt 注入配置，不是完整 Skill。全局配置中展示“全局分析策略”，特性详情中展示“特性分析策略”。策略表单包含名称、适用阶段、优先级、启用开关和 Prompt 内容；创建、编辑、删除、启停都使用统一列表样式。未来如果扩展真正 Skill Package，再单独设计工具能力、触发条件、示例和版本管理。

全局壳层统一为：

```text
TopBar: Logo + CodeAsk                                      UserMenu
Sidebar: 会话 / 特性 / 设置
Main: 当前页面
```

TopBar 左上角只保留 Logo 和 `CodeAsk`，不再展示"研发知识工作台"描述。右上角用户菜单展示当前自报身份或管理员身份，并预置登录、个人信息、设置、退出入口。

一级侧边栏和所有二级侧边栏都必须支持展开 / 收起。一级侧边栏收起后保留图标导航；会话、特性和设置里的二级侧边栏收起后保留窄条与恢复按钮，不改变当前页面上下文。收起 / 展开控件放在侧边栏与右侧内容交界的中部，不放在侧边栏头部；鼠标悬停时使用轻量浮动和箭头位移动效表达收起方向，并尊重 `prefers-reduced-motion`。

前端必须自托管中文字体 fallback，避免私有部署环境或 Playwright/CI 容器缺少 CJK 系统字体时出现中文方块。

## 3. 会话工作台

建议三栏布局：

```text
左侧                中间                右侧
─────              ──────              ──────
会话列表            聊天流              调查进度面板
搜索 + 新建         输入区              工具调用流
三点菜单            操作行              证据列表
批量选择            上传日志/生成报告    ask_user 区
                                      A2 定界结果
                                      A3 充分性判断
                                      会话数据
```

### 3.1 会话列表

默认按当前 `subject_id` 过滤显示当前用户自己的会话（详见 §8 自报身份），不提供"我的/全部"切换。

管理员调阅其他人的会话属于后续审计/协作能力，不在当前 workbench 主界面暴露。

会话条目展示：

- 标题（自动从首条提问生成，可改）
- 关联特性
- 最近反馈状态（已解决 / 部分解决 / 没解决 / 待反馈）
- 右侧三点菜单：编辑名称、分享、置顶、批量操作、删除

三点菜单必须脱离列表卡片 / 滚动容器裁切，确保完整下拉项可见。

无会话时输入框仍可用。用户首次发送消息时，前端自动创建默认会话并把消息发送到该会话。

批量操作进入选择模式，列表项前显示 checkbox，并提供批量删除。

打开或刷新已有会话时，前端必须通过 `GET /api/sessions/{id}/turns` 拉取持久化的 `session_turns` 并渲染聊天流，同时通过 `GET /api/sessions/{id}/traces` 拉取 `agent_traces` 回放右侧调查进度和运行事件。会话列表只代表会话元数据存在，不代表消息或运行轨迹已经在本地状态中；历史恢复不能依赖 SSE 直播期间追加到 React state 的临时数据。

### 3.2 输入区

用户提问**不需要**选场景、选类型——Agent 自动定界（PRD §4.2）。

输入区支持：

- 自然语言文本
- 附件上传（日志文件、截图、文档片段）
- 粘贴日志（自动识别 + 抽取关键信号）

可选预填字段（详见 `debugging-workflow.md` §2，全部非必填）：environment / service / time_range / impact / recent_changes / version_hint。

上传日志不依赖已有会话。无会话时点击上传并选择文件，前端自动创建默认会话（标题默认取文件名）并把附件写入该会话。上传按钮不能因为当前没有会话而禁用。

### 3.3 会话数据与附件隔离

每个会话都有独立的临时数据目录，路径形态为：

```text
<data_dir>/sessions/<session_id>/
```

用户在某个会话上传的日志、截图和文档片段只属于该会话；切换到另一个会话时，右侧“会话数据”区域必须重新读取目标会话的附件列表，不得显示上一会话的数据。

同名文件允许重复上传。典型场景是多节点数据库或服务集群，每个节点都上传 `service.log`。后端物理文件名使用唯一 `attachment_id` 保存，前端展示：

- 可编辑的 `display_name`
- 原始文件名 `original_filename`
- 历史别名 `aliases`
- 用户补充的用途说明 `description`
- 附件短 ID、大小、类型和上传时间

会话数据区域支持：

- 列表查看已上传数据
- 重命名展示名
- 编辑用途说明，用于记录“节点 A 的数据库日志”“主库慢查询日志”等口语化映射
- 删除附件；删除成功后后端移除 DB 记录并尽力删除物理文件

附件编辑只修改会话附件元数据，不改变用户原始文件名和物理文件名。后端必须保留历史别名，并以 `attachment_id` 作为 AI 和工具读取日志时的稳定引用；删除附件只影响当前会话。重命名、编辑说明和删除成功后的提示是瞬时反馈，必须自动消失，避免会话界面长期残留过期状态。

## 4. 右侧调查面板

调查面板让用户看到 Agent 正在做什么、做到哪里。

历史会话同样必须展示调查面板。刷新或切换会话后，右侧面板从 `/api/sessions/{id}/traces` 重建阶段状态；运行事件展示范围判断、充分性判断、工具调用、工具结果和证据摘要。工具事件只能显示短摘要，例如 `query=litellm`、`成功 · 24 code matches · 24 条命中`，不得把完整 JSON payload 直接铺到界面里。

### 4.1 阶段透明（与 `agent-runtime.md` §11 SSE 事件对应）

```text
[当前阶段] CodeInvestigation
  └ "知识库不足，正在使用默认分支扫 order-service"

工具调用流：
  ✓ search_wiki(订单 ERR_ORDER_CONTEXT_EMPTY)
  ✓ read_wiki_doc(《支付回调》#3)
  ⟳ grep_code(payment_method)
```

### 4.2 A2 定界透明（PRD §7.1.A2 缓解措施）

定界结果区显著展示：

```text
Agent 选择了特性：[订单] ▼ ← 一键切到候选列表
信心：high
理由：日志中出现 OrderService 符号
```

下拉里展示 Agent 给出的候选特性 + confidence，用户可一键改正。

如果 Agent confidence 为 low 或多个候选概率接近 → 自动触发 `ask_user`（详见 `agent-runtime.md` §4），不直接进入检索。

### 4.3 A3 充分性判断透明（PRD §7.1.A3 缓解措施）

接收 `sufficiency_judgement` SSE 事件后展示：

```text
[充分性判断] insufficient
  └ "已查 4 篇文档，但均未涉及字段变更影响清单"

  下一步：进入 CodeInvestigation

  [再深查一下] ← 用户兜底按钮（强制进入代码层）
```

即使 Agent 判定"够"，UI 也保留"再深查一下"按钮——让用户在不满意答案时强制扩展调查（PRD §7.1.A3 缓解措施）。

### 4.4 版本状态

代码证据来源版本必须可见：

```text
代码版本：order-service @ default branch (探索性预查)
          ⚠ 未确认故障版本，仅作初步判断
```

详见 `code-index.md` §5 默认分支预查。

## 5. 答案展示（PRD §8.4 落地）

回答按"结论先 + 证据折叠"组织。oncall 凌晨 2 点要能 5 秒看懂。

Agent 回答、历史消息和问题报告详情都按 Markdown 渲染，支持标题、列表、表格、引用和代码块。聊天气泡中不得直接显示 Markdown 源码标记（例如裸露的 `# 标题`），除非这些符号位于代码块内。代码块需要提供 hover 复制按钮，并保证中文和英文混排时字体 fallback 正常显示。

消息气泡 hover 或键盘聚焦时，在气泡底部浮现紧凑 icon 操作条。当前至少包含复制、重新生成、分享；暂不支持的动作点击后给出瞬时提示，不阻塞会话。复制操作使用完整消息文本，不复制渲染后的 DOM 片段。

### 5.1 默认结构

```text
╔═══════════════════════════════════════════╗
║ [结论]                          置信度: medium ║
║ payment-gateway 调用 timeout 是 hardcode 1s   ║
║                                              ║
║ [建议操作]                                   ║
║ • 把 timeout 临时调大（例 3s）                ║
║ • 联系 payment-gateway 团队看服务健康         ║
║ ※ 这是建议，决策权在你                        ║
║                                              ║
║ [不确定点]                                   ║
║ • 没有 payment-gateway 自身日志               ║
║ • 没看到流量 / QPS 数据                       ║
║                                              ║
║ [反馈] ✓ 已解决  ◐ 部分解决  ✗ 没解决          ║
║                                              ║
║ ▼ 证据（折叠）                               ║
╚═══════════════════════════════════════════╝
```

### 5.2 结论区

- 结论文本醒目（大字号、高对比度）。
- 置信度标签：`high` / `medium` / `low`，配色区分。
- 与 `evidence-report.md` §4 结论模型一一对应。

### 5.3 建议操作区

- 列出可执行步骤。
- **必须**带"决策权在你"提示——回答**绝不能**写成"按这样改就行了"（PRD §6.1 不假装自信）。

### 5.4 不确定点区

- Agent 主动暴露的盲区。
- 这是诚实的体现——空着不写比写满更危险。

### 5.5 证据区

默认折叠。展开后展示：

- 日志行号 + 原文片段
- 文档路径 + heading + 段落引用
- 报告链接（带"已验证 / draft"标识）
- 代码：repo + path + 行号 + commit sha
- 如果代码来自默认分支，每条代码证据醒目标注"未确认故障版本"

## 6. 反馈与 deflection rate（PRD §5.1）

每个回答下方的反馈按钮是 deflection rate 双轨测量的目标交互（PRD §5.1）。当前实现已经接入：

- `POST /api/feedback`：保存显式反馈。
- `POST /api/events`：保存 `feedback_submitted`、`force_deeper_investigation` 等前端事件。

Dashboard 聚合仍是后续前端增强，不影响 raw data 写入。

### 6.1 显式反馈

| 按钮 | 含义 | 写入 |
|---|---|---|
| ✓ 已解决 | 不需要再问真人 | `feedback = "solved"`, deflection +1 |
| ◐ 部分解决 | 拿到方向但需要补充 | `feedback = "partial"`, 进入 Maintainer dashboard 关注 |
| ✗ 没解决 / 答错 | 系统无效 / 错误 | `feedback = "wrong"`, 进入护栏指标（PRD §5.2 错误反馈率） |

每个按钮可附备注（"漏了 settlement-service 里的反射调用"）。

### 6.2 隐式反馈

- 用户在 N 分钟内未产生新提问、未点"找人帮忙" → 视为"自然结束"
- 与显式反馈相互校准（PRD §5.1）

### 6.3 找人帮忙（一期不内置）

PRD §8.4 末段："找哪个团队帮忙一期不内置，但知识库里如果有'团队 owner / 联系方式'文档，Agent 就能引用"。

UI 不提供"找人"按钮——避免 deflection 数据被"系统帮你找人"污染。如果用户去找真人，靠隐式反馈（未自然结束）捕捉。

## 7. 报告操作

回答后提供：

- 生成报告草稿（Agent 预填问题 / 回答 / 证据 / 用户备注，详见 `evidence-report.md` §6.1）。该入口位于会话输入区操作行，放在"强制代码调查"右侧、"发送"左侧。
- 标记有用 / 部分解决 / 错误（用于 deflection rate 测量，与 §6 一致；当前通过 `/api/feedback` 和 `/api/events` 持久化 raw data）。
- 一键验证（一期任意人可验证，详见 `evidence-report.md` §7）。
- 下载报告。

生成报告前必须做初步确认：会话至少完成一次用户提问和 Agent 回答，否则弹出"暂不能生成报告"提示，不调用报告 API。满足条件后弹出确认框，要求绑定特性；如果 Agent 通过 `scope_detection.feature_ids` 推断出特性，则默认选中该特性，否则由用户选择。生成成功后给出"查看报告"入口，跳转到绑定特性的"问题报告" tab 并选中刚生成的报告。

特性页的问题报告 tab 只展示由会话生成或归档到该特性的报告，不提供手工创建报告入口。报告列表顶部提供状态筛选标签，至少包含"全部"、"草稿"、"已验证"、"未通过"，并按报告 `status` 展示对应状态徽标；后续出现 `stale`、`superseded` 等状态时应显示为独立标签或清晰徽标，不得把非 verified 状态统一渲染成"草稿"。

报告详情使用 Markdown 渲染。右侧预览区必须被限制在当前详情面板内并拥有独立纵向滚动条：长报告不能撑破页面边界；长路径、长英文 token、代码块和表格只能在预览区内部换行或横向滚动。

报告详情页额外提供：

- "编辑报告"：内联编辑标题和 Markdown 正文，仅保存当前报告内容，不在特性页创建新报告。
- "删除报告"：删除前必须确认，成功后从当前列表和详情预览中移除。
- "验证通过"：将报告从 `draft` 或 `rejected` 标记为 `verified`，并进入报告检索索引。
- "验证不通过"：将报告标记为 `rejected`，不进入报告检索索引。
- 显著展示 `verified_by`（自报身份 + "自报"标识）和 `verified_at`。
- "撤销验证"按钮：一键回退到 `draft`，从索引中下架（PRD §4.4.1 污染缓解措施）。

一期**不含**"提交验证"按钮——PRD §6.2 MVP Anti-Goal 不做审核工作流，`draft → verified` 是一步。

## 8. 用户菜单、自报身份与管理员入口

一期采用"自报身份"软识别（PRD §4.2 / §4.4.1，详见 `deployment-security.md` §3）。

### 8.1 首次访问

```text
检测到 localStorage 无 client_id
→ 自动生成 UUID 写入 localStorage
→ 弹窗（非阻塞，可跳过）："你想给自己起个昵称吗？方便在团队里区分会话所有者"
→ 用户输入昵称 → 写入 localStorage
→ 用户跳过 → 显示为"设备 X-<short_id>"
```

整个流程不阻断使用，跳过的用户立即可以提问。

### 8.2 用户菜单

顶栏 `<UserMenu />` 一期默认实现：

- 未登录时只显示"登录"菜单项，点击进入中性登录页
- 管理员登录后显示管理员身份
- 登录后菜单包含个人信息、设置、退出
- 个人信息当前阶段显示"暂不支持"

登录页在认证前使用中性文案：标题为"登录"，错误提示只说明用户名或密码不正确，不提前暴露管理员权限含义。密码输入框右侧提供眼睛图标用于明文 / 密文切换。

未来接入 AuthProvider 时，此 slot 由实际 Provider 的实现替换（`OidcUserMenu` / `LdapUserMenu` 等，详见 `deployment-security.md` §4.2）。

### 8.3 会话归属

会话工作台（§3）左侧会话列表按当前 `subject_id` 过滤显示当前用户的会话。

当前会话在消息区标题右侧展示短 ID 椭圆标签，例如 `sess_71b4`。短 ID 取真实 `session_id` 的前几位用于视觉定位；点击标签复制完整 `session_id`，并在标签旁展示 1 秒的复制成功提示，方便用户把页面中的会话和磁盘上的 `sessions/<session_id>/` 目录直接对应起来。

### 8.4 不做的事

为避免假装是鉴权，UI 明确**不做**：

- 不强制昵称唯一（两人都填"Alice"允许；提示但不阻止）
- 不阻止改昵称 / 清身份 / 切换 client_id
- 不把自报身份当作真实安全边界

当前阶段已经引入最小管理员边界：普通用户仍可匿名访问并只看到自己的会话和个人配置；管理员设置页只展示全局配置，不展示个人用户配置；全局 LLM 配置、全局仓库写操作和后续系统级设置只对管理员可见。自报身份仍不是企业级鉴权，真正的账号体系留给 `deployment-security.md` §4 的 AuthProvider 扩展。

## 9. Maintainer Dashboard（后续增强设计）

PRD §4.2：

> 提供 dashboard 让 Maintainer 看到飞轮关键信号：被标"部分解决"或"答错"的会话、待审核的报告草稿、知识库文档变更等。无 dashboard → 飞轮转不起来。

当前 frontend-workbench 阶段不把 Dashboard 作为验收阻塞项。Dashboard 的真实数据源来自已落地的 `feedback`、`frontend_events`、`audit_log` 以及已有 `agent_traces`；后续需要在这些 raw events 之上补聚合查询和专用前端页面。下面保留的是后续增强设计。

### 9.1 默认展示

```text
┌─ 待我处理 ───────────────────────────────────┐
│ • 报告草稿 3 条等待验证                        │
│ • "部分解决"会话 5 条（最近 7 天）              │
│ • "答错"会话 1 条 ⚠                            │
└──────────────────────────────────────────────┘

┌─ 飞轮信号（最近 30 天）───────────────────────┐
│ deflection rate: 42% (↑ 5pp)                   │
│ 错误反馈率: 3% (持平)                          │
│ 新增已验证报告: 4 条                            │
│ 已验证报告被命中: 17 次                         │
│ 知识库文档新增 / 更新: 6 条                     │
└──────────────────────────────────────────────┘

┌─ 关注的特性 ─────────────────────────────────┐
│ [订单] [结算] ...（用户自己勾选）              │
│ 各特性的提问数 / deflection / 错误反馈率       │
└──────────────────────────────────────────────┘
```

### 9.2 关键交互

- 点击"部分解决"会话条目 → 跳转到该会话 → 一键"沉淀为报告草稿"（PRD §8.3 旅程 3）。
- 点击"报告草稿"条目 → 报告详情页 → 编辑 + 验证或撤销。
- 点击"答错"会话 → 跳转 → 让 Maintainer 判断是回答错误（标 wrong）还是问题歧义。

### 9.3 一期不做

为避免范围膨胀：

- 不做 Maintainer 实时通知（邮件 / IM）——dashboard 是拉取模型，不主动推送
- 不做团队级聚合——仅当前用户视角（即"我关注的特性"）
- 不做趋势图细分到日——按 7 天 / 30 天聚合即可
