# 前端研发工作台设计

> 本文档属于 v1.0 SDD，描述前端工作台的页面结构与交互。
>
> 产品契约见同版本 `prd/codeask.md`。当 SDD 与 PRD 冲突时，以 PRD 为准。

## 1. 目标

前端不是单纯聊天框，而是研发调查工作台。用户需要看到系统正在查什么、找到什么证据、当前结论是否受版本限制。

落地 PRD 的几条关键 UI 承诺：

- **结论先 + 证据折叠**（PRD §8.4）：oncall 凌晨 2 点要能 5 秒看懂结论。
- **A2 / A3 透明**（PRD §7.1）：定界结果和"知识库够不够"判断必须可见，并提供兜底按钮。
- **deflection rate 反馈**（PRD §5.1）：每个回答下方有"已解决 / 部分解决 / 没解决"按钮。
- **Maintainer dashboard**（PRD §4.2）：让 Maintainer 看到飞轮关键信号——无 dashboard 飞轮转不起来。
- **不假装自信**（PRD §6.1）：建议操作要明确"是建议、决策权在人"。

## 2. 页面结构

一期页面：

| 页面 | 用途 |
|---|---|
| 会话工作台 | 提问、调查可视化、答案展示（§3-§6） |
| Maintainer Dashboard | 飞轮关键信号集中视图（§9） |
| Wiki 管理 | 文档、报告管理 |
| 特性详情 | 特性元信息、关联仓库勾选、特性 skill |
| 仓库配置 | 全局仓库池注册（详见 `code-index.md` §2） |
| Skill 管理 | 全局 skill / 特性 skill |
| LLM 配置 | 模型供应商、API Key |
| 报告详情 | 单份报告详情、验证操作（§7） |

## 3. 会话工作台

建议三栏布局：

```text
左侧                中间                右侧
─────              ──────              ──────
会话列表            聊天流              调查进度面板
（默认"我的"）      输入区              工具调用流
"全部会话"切换      附件区              证据列表
新建会话                                ask_user 区
特性筛选                                A2 定界结果
                                       A3 充分性判断
```

### 3.1 会话列表

默认按当前 `subject_id` 过滤显示"我的会话"（详见 §8 自报身份）。

提供"全部会话"切换视图——这是飞轮所必需（如 Maintainer 看 Dave 的 oncall 会话决定沉淀为报告，PRD §8.4）。

会话条目展示：

- 标题（自动从首条提问生成，可改）
- 创建者 `subject_id`（带"自报"标识）
- 关联特性
- 最近反馈状态（已解决 / 部分解决 / 没解决 / 待反馈）

### 3.2 输入区

用户提问**不需要**选场景、选类型——Agent 自动定界（PRD §4.2）。

输入区支持：

- 自然语言文本
- 附件上传（日志文件、截图、文档片段）
- 粘贴日志（自动识别 + 抽取关键信号）

可选预填字段（详见 `debugging-workflow.md` §2，全部非必填）：environment / service / time_range / impact / recent_changes / version_hint。

## 4. 右侧调查面板

调查面板让用户看到 Agent 正在做什么、做到哪里。

### 4.1 阶段透明（与 `agent-runtime.md` §11 SSE 事件对应）

```text
[当前阶段] CodeInvestigation
  └ "知识库不足，正在使用默认分支扫 order-service"

工具调用流：
  ✓ search_wiki(订单 ERR_ORDER_CONTEXT_EMPTY)
  ✓ read_wiki_doc(《支付回调》#3)
  ⟳ grep_code(payment_method)
```

### 4.2 A2 定界透明（PRD §7.1.A2 缓解措施）

定界结果区显著展示：

```text
Agent 选择了特性：[订单] ▼ ← 一键切到候选列表
信心：high
理由：日志中出现 OrderService 符号
```

下拉里展示 Agent 给出的候选特性 + confidence，用户可一键改正。

如果 Agent confidence 为 low 或多个候选概率接近 → 自动触发 `ask_user`（详见 `agent-runtime.md` §4），不直接进入检索。

### 4.3 A3 充分性判断透明（PRD §7.1.A3 缓解措施）

接收 `sufficiency_judgement` SSE 事件后展示：

```text
[充分性判断] insufficient
  └ "已查 4 篇文档，但均未涉及字段变更影响清单"
  
  下一步：进入 CodeInvestigation
  
  [再深查一下] ← 用户兜底按钮（强制进入代码层）
```

即使 Agent 判定"够"，UI 也保留"再深查一下"按钮——让用户在不满意答案时强制扩展调查（PRD §7.1.A3 缓解措施）。

### 4.4 版本状态

代码证据来源版本必须可见：

```text
代码版本：order-service @ default branch (探索性预查)
          ⚠ 未确认故障版本，仅作初步判断
```

详见 `code-index.md` §5 默认分支预查。

## 5. 答案展示（PRD §8.4 落地）

回答按"结论先 + 证据折叠"组织。oncall 凌晨 2 点要能 5 秒看懂。

### 5.1 默认结构

```text
╔═══════════════════════════════════════════╗
║ [结论]                          置信度: medium ║
║ payment-gateway 调用 timeout 是 hardcode 1s   ║
║                                              ║
║ [建议操作]                                   ║
║ • 把 timeout 临时调大（例 3s）                ║
║ • 联系 payment-gateway 团队看服务健康         ║
║ ※ 这是建议，决策权在你                        ║
║                                              ║
║ [不确定点]                                   ║
║ • 没有 payment-gateway 自身日志               ║
║ • 没看到流量 / QPS 数据                       ║
║                                              ║
║ [反馈] ✓ 已解决  ◐ 部分解决  ✗ 没解决          ║
║                                              ║
║ ▼ 证据（折叠）                               ║
╚═══════════════════════════════════════════╝
```

### 5.2 结论区

- 结论文本醒目（大字号、高对比度）。
- 置信度标签：`high` / `medium` / `low`，配色区分。
- 与 `evidence-report.md` §4 结论模型一一对应。

### 5.3 建议操作区

- 列出可执行步骤。
- **必须**带"决策权在你"提示——回答**绝不能**写成"按这样改就行了"（PRD §6.1 不假装自信）。

### 5.4 不确定点区

- Agent 主动暴露的盲区。
- 这是诚实的体现——空着不写比写满更危险。

### 5.5 证据区

默认折叠。展开后展示：

- 日志行号 + 原文片段
- 文档路径 + heading + 段落引用
- 报告链接（带"已验证 / draft"标识）
- 代码：repo + path + 行号 + commit sha
- 如果代码来自默认分支，每条代码证据醒目标注"未确认故障版本"

## 6. 反馈与 deflection rate（PRD §5.1）

每个回答下方有反馈按钮，用于 deflection rate 双轨测量（PRD §5.1）。

### 6.1 显式反馈

| 按钮 | 含义 | 写入 |
|---|---|---|
| ✓ 已解决 | 不需要再问真人 | `feedback = "solved"`, deflection +1 |
| ◐ 部分解决 | 拿到方向但需要补充 | `feedback = "partial"`, 进入 Maintainer dashboard 关注 |
| ✗ 没解决 / 答错 | 系统无效 / 错误 | `feedback = "wrong"`, 进入护栏指标（PRD §5.2 错误反馈率） |

每个按钮可附备注（"漏了 settlement-service 里的反射调用"）。

### 6.2 隐式反馈

- 用户在 N 分钟内未产生新提问、未点"找人帮忙" → 视为"自然结束"
- 与显式反馈相互校准（PRD §5.1）

### 6.3 找人帮忙（一期不内置）

PRD §8.4 末段："找哪个团队帮忙一期不内置，但知识库里如果有'团队 owner / 联系方式'文档，Agent 就能引用"。

UI 不提供"找人"按钮——避免 deflection 数据被"系统帮你找人"污染。如果用户去找真人，靠隐式反馈（未自然结束）捕捉。

## 7. 报告操作

回答后提供：

- 生成报告草稿（Agent 预填问题 / 回答 / 证据 / 用户备注，详见 `evidence-report.md` §6.1）。
- 标记有用 / 部分解决 / 错误（用于 deflection rate 测量，与 §6 一致）。
- 一键验证（一期任意人可验证，详见 `evidence-report.md` §7）。
- 下载报告。

报告详情页额外提供：

- 显著展示 `verified_by`（自报身份 + "自报"标识）和 `verified_at`。
- "撤销验证"按钮：一键回退到 `draft`，从索引中下架（PRD §4.4.1 污染缓解措施）。

一期**不含**"提交验证"按钮——PRD §6.2 MVP Anti-Goal 不做审核工作流，`draft → verified` 是一步。

## 8. 用户菜单与自报身份

一期采用"自报身份"软识别（PRD §4.2 / §4.4.1，详见 `deployment-security.md` §3）。

### 8.1 首次访问

```text
检测到 localStorage 无 client_id
→ 自动生成 UUID 写入 localStorage
→ 弹窗（非阻塞，可跳过）："你想给自己起个昵称吗？方便在团队里区分会话所有者"
→ 用户输入昵称 → 写入 localStorage
→ 用户跳过 → 显示为"设备 X-<short_id>"
```

整个流程不阻断使用，跳过的用户立即可以提问。

### 8.2 用户菜单

顶栏 `<UserMenu />` 一期默认实现：

- 显示当前 `subject_id` + "自报"小标识
- "改昵称"入口（任意时刻可改，立即生效）
- "清除身份"入口（清空 localStorage，下次访问重新生成）

未来接入 AuthProvider 时，此 slot 由实际 Provider 的实现替换（`OidcUserMenu` / `LdapUserMenu` 等，详见 `deployment-security.md` §4.2）。

### 8.3 会话归属

会话工作台（§3）左侧会话列表默认按 `subject_id` 过滤显示"我的会话"。

切换"全部会话"视图后可见所有用户的会话——这是飞轮所必需（如 Maintainer 看 Dave 的 oncall 会话决定沉淀为报告，PRD §8.4）。

### 8.4 不做的事

为避免假装是鉴权，UI 明确**不做**：

- 不强制昵称唯一（两人都填"Alice"允许；提示但不阻止）
- 不阻止改昵称 / 清身份 / 切换 client_id
- 不基于身份做访问控制（任何人仍可看任何会话 / 报告 / 文档）

以上保持 PRD §4.4.1"无身份模型 + 物理隔离"的取舍不变。

## 9. Maintainer Dashboard（PRD §4.2 关键承诺）

PRD §4.2：

> 提供 dashboard 让 Maintainer 看到飞轮关键信号：被标"部分解决"或"答错"的会话、待审核的报告草稿、知识库文档变更等。无 dashboard → 飞轮转不起来。

### 9.1 默认展示

```text
┌─ 待我处理 ───────────────────────────────────┐
│ • 报告草稿 3 条等待验证                        │
│ • "部分解决"会话 5 条（最近 7 天）              │
│ • "答错"会话 1 条 ⚠                            │
└──────────────────────────────────────────────┘

┌─ 飞轮信号（最近 30 天）───────────────────────┐
│ deflection rate: 42% (↑ 5pp)                   │
│ 错误反馈率: 3% (持平)                          │
│ 新增已验证报告: 4 条                            │
│ 已验证报告被命中: 17 次                         │
│ 知识库文档新增 / 更新: 6 条                     │
└──────────────────────────────────────────────┘

┌─ 关注的特性 ─────────────────────────────────┐
│ [订单] [结算] ...（用户自己勾选）              │
│ 各特性的提问数 / deflection / 错误反馈率       │
└──────────────────────────────────────────────┘
```

### 9.2 关键交互

- 点击"部分解决"会话条目 → 跳转到该会话 → 一键"沉淀为报告草稿"（PRD §8.3 旅程 3）。
- 点击"报告草稿"条目 → 报告详情页 → 编辑 + 验证或撤销。
- 点击"答错"会话 → 跳转 → 让 Maintainer 判断是回答错误（标 wrong）还是问题歧义。

### 9.3 一期不做

为避免范围膨胀：

- 不做 Maintainer 实时通知（邮件 / IM）——dashboard 是拉取模型，不主动推送
- 不做团队级聚合——仅当前用户视角（即"我关注的特性"）
- 不做趋势图细分到日——按 7 天 / 30 天聚合即可

## 10. 与 PRD 的对齐

本文已按 `prd/codeask.md` §9 对齐表 + §4.2 + §5.1 + §7.1 + §8.4 对应承诺更新，主要变化：

- §4 右侧调查面板新增 A2 定界透明（4.2）和 A3 充分性判断透明（4.3）+ "再深查一下"兜底按钮，落地 PRD §7.1 致命假设缓解措施
- §5 答案展示按 PRD §8.4 重写（结论醒目 / 建议操作 / 不确定点 / 证据折叠 / 不假装自信）
- §6 新增反馈机制（显式 + 隐式 + "找人帮忙"取舍），落地 PRD §5.1 deflection rate 双轨测量
- §7 报告操作删除"提交验证"按钮（PRD §6.2 不做审核工作流），加"撤销验证"和身份显示
- §8 自报身份完整流程（首次访问 / 用户菜单 / 会话归属 / 不做的事），落地 PRD §4.2 / §4.4.1
- §9 新增 Maintainer Dashboard 完整设计（待处理 / 飞轮信号 / 关注特性），落地 PRD §4.2"无 dashboard 飞轮转不起来"承诺
- §3 会话工作台明确"我的会话 / 全部会话"双视图，兼顾默认有序 + 飞轮所需开放访问
