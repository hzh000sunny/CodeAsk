## §1 产品定位

**CodeAsk 是一个让团队成员自助查到答案、让资深工程师把脑中知识低成本沉淀下来的私有部署研发问答系统。**

它面向需要私有部署的研发团队，把团队的代码、文档和已验证经验组织为可检索的知识库，让 AI 基于团队的真实知识——既让提问者随时拿到带证据的可信回答，也让贡献者不必反复口头回答同一问题。

CodeAsk **不是**：

- 公共代码助手（Copilot、Cursor 类）
- 面向单人使用的 AI Pair Programmer
- 无知识库支撑的"零样本代码理解"工具

**核心价值主张**：

> "知识不再卡在某个人脑子里。"

这一句对两端都成立：

- 对**贡献者**（如资深工程师）：脑中知识能低成本沉淀，不用反复口头答疑
- 对**提问者**（新人 / 跨团队 / oncall）：随时拿到带证据的可信回答，不用打扰真人

## §2 核心画像

CodeAsk 围绕**两类同等重要的使用者**设计。两者不是"主-次"，而是飞轮的两端，缺一端飞轮就停。

### 2.1 角色 A — "被问的资深工程师" (Maintainer)

| 维度 | 内容 |
|---|---|
| 痛点 | 白天被同事 / 新人持续问关于自己模块的问题，被打断"写代码、做架构决策"的高价值工作 |
| 在 CodeAsk 中 | 知识库的**贡献者** —— 注册仓库、建特性、上传知识库、审核报告 |
| 产品对其承诺 | 低摩擦地把脑中知识沉淀下来；不需要再面对面回答同一个问题第 N 次 |
| 成功标准 | 被问同一类问题的次数下降；维护知识库的时间 < 节省下来的被打扰时间 |

### 2.2 角色 B — "提问者" (Asker)

| 维度 | 内容 |
|---|---|
| 来源 | 团队新人、跨模块协作者、跨团队请教者、oncall 救火者 |
| 痛点 | 不熟某模块、不知该问谁、文档分散难找、Slack/IM 历史搜不到 |
| 在 CodeAsk 中 | 知识库的**消费者** —— 自然语言提问，期待带证据的可信回答 |
| 产品对其承诺 | 随时拿到带证据的可信回答；不必去打扰一个真人 |
| 成功标准 | 回答能解决问题、可信、不用再去找人 |

### 2.3 飞轮约束

任一端的体验崩了，整个产品就崩：

| 这一端体验差 | 直接后果 | 飞轮影响 |
|---|---|---|
| Maintainer 录入太麻烦 | 没人维护知识库 | 知识库变空 → Asker 没东西可查 |
| Asker 回答不可信 | 还得去找真人 | deflection rate 不上升 → Maintainer 觉得没用、不再维护 |

PRD 后续每个功能决策都要问两个问题：

1. **这对 Maintainer 录入摩擦影响多大？**
2. **这对 Asker 回答可信度影响多大？**

任一端被牺牲都不行。

## §3 主链路

```text
用户提问（任意自然语言，无需选场景或类型）
  ↓
Agent 自动定界到特性
  ├─ 命中明确特性 → 锁定该特性下的知识库 + 代码仓
  └─ 定界失败 → 追问用户 OR 全局检索降级
  ↓
知识库优先检索
  ├─ 已验证报告（高优先级 — 因为人审过）
  ├─ 特性下的文档/规范/排障手册
  └─ 全局文档（兜底）
  ↓
[Agent 判断知识库是否足够回答]
  ↓ 不够
代码深入分析
  ├─ git worktree 方式把特性关联的代码仓挂载到会话临时空间
  ├─ Agent 用 grep / read_file / list_symbols 等工具
  └─ 具体分析什么由 Agent 按用户描述自适应（不预设 query 形态）
  ↓
答案合成 + 证据引用
  ├─ 简单问答 → 文档级证据
  └─ 复杂问答 → 文档证据 + 代码证据 + commit 绑定
  ↓
（可选）生成报告草稿 → 任意团队成员审核（社会契约约定 owner 来做）→ 验证后入库
```

### 3.1 关键约束

- 用户**必须先**在特性下建知识库，否则该特性下的提问无法处理（详见 §4.1）
- Agent **自动判断**"知识库够不够"是产品的核心智能假设（详见 §7.1.A3）
- 代码层访问按**会话隔离**（多人多会话不互相干扰，详见 §4.2）
- 跟同版本 `design/overview.md` 中的原描述不同：**主链路只有两层（知识库 → 代码）**，不再单独把"已验证报告"作为一层。报告作为知识库内的高优先级文档存在。

### 3.2 三个原本拆成 P0 / P1 / P2 的场景在新主链路下的归并

| 场景 | 在主链路中的走向 |
|---|---|
| 知识库问答（原 P2） | 简单问题，知识库层就结束 |
| 代码理解（原 P1） | 知识库不够 → 进代码层；问题复杂度决定深入程度 |
| 日志排障（原 P0） | 复杂问题的特例（query 是日志 + 现象）；走代码层；事后沉淀报告 |

它们不再是优先级有别的三件事，而是**同一主链路在不同复杂度上的走向**。

## §4 产品契约

### 4.1 用户侧约定（准入门槛）

| 约定 | 说明 |
|---|---|
| 必须先建特性 | 没有特性就没有问答；特性是检索的边界、责任的边界 |
| 必须为每个特性上传知识库 | 至少一份基础说明 / 设计文档 / 接口说明，否则该特性的提问无法处理 |
| 必须为特性勾选关联代码仓 | 没有代码仓只能做文档级回答，不能做代码级深入分析 |
| 必须有人审核报告 | 没有人审核 → 报告永远是 draft → 飞轮转不起来；审核者由特性 owner 承担（一期为社会契约约定，非系统强制） |

**注意**：特性粒度不强制。可以是业务模块、可以是微服务、可以是任意"有 owner 的知识集合"——用户自定义。特性之间可以毫无关联。

### 4.2 产品侧承诺

| 承诺 | 说明 |
|---|---|
| 全局仓库池 | 配置页填写仓库名称 + URL，加完后端**异步下载到公共目录**，用户无感等待 |
| 特性勾选式关联仓库 | 在特性管理页勾选要关联的仓库，从全局池选择 |
| 会话级代码访问隔离 | 使用 **git worktree** 把代码仓挂载到会话独立空间，多人多会话不互相干扰；轻量、共享 git 数据、不重复占磁盘 |
| Agent 自动定界 | 用户提问时不需要选场景、选类型，Agent 基于问题描述 + 各特性摘要自动选择 |
| Agent 自动路由复杂度 | 用户不感知"知识库够不够"——Agent 决定何时进入代码层 |
| 回答必须带证据 | 简单问答 → 文档引用；复杂问答 → 文档 + 代码片段 + commit 绑定 |
| 不确定就坦白 | 答不出 / 证据不足 / 多个候选根因 → 系统必须明示，不许"装作答出来" |
| 多轮会话与历史恢复 | 同一会话内多轮提问，上下文（特性、版本、已检索结果、历史问答、已确认信息）持续复用；用户可恢复历史会话继续讨论 |
| 自报身份与会话归属 | 一期会话绑定到"自报身份"（client_id + 可选昵称，存 localStorage），UI 默认显示"我的会话"；这是软识别非鉴权 |
| 会话开放访问 | 任何人仍可在"全部会话"视图打开他人会话（用于飞轮沉淀，如 Maintainer 沉淀 oncall 案例为报告） |
| Maintainer 关注列表 | 提供 dashboard 让 Maintainer 看到飞轮关键信号：被标"部分解决"或"答错"的会话、待审核的报告草稿、知识库文档变更等。无 dashboard → 飞轮转不起来 |
| 私有部署 | 默认本机监听；数据 / 索引 / 会话不出本机 |

### 4.3 PRD 阶段不锁死的事

以下是 SDD / 实现阶段才确定的细节，PRD 不锁：

- LLM 选哪个供应商（OpenAI / Anthropic / 本地模型 / OpenAI-compatible）
- 知识库索引的具体技术（FTS5 / 向量 / 混合检索）
- 数据库选型（SQLite / Postgres）
- 异步 clone 的具体实现（subprocess / aiohttp / git library）

### 4.4 权限模型

#### 4.4.1 一期（MVP）—— 无身份模型

| 项 | 一期决策 |
|---|---|
| 登录 | **无登录** |
| 鉴权 | **无鉴权**（监听 127.0.0.1 + 私有部署 = 物理隔离） |
| 用户身份 | 系统不区分谁是谁；所有人都是匿名访问 |
| 知识库读写权限 | **所有人可读、所有人可写**（含修改特性、上传文档、删除报告） |
| 报告审核 | **任何人都能验证报告**——不通过权限强制，而通过团队**社会契约**约束（"特性 owner"是命名上的角色，不是系统强制） |

**取舍说明**：放弃权限隔离换取**极低的部署门槛和上手成本**。一个新部署的团队 30 秒内能让所有人开始用，不需要先做账号体系对接——这对"任意团队都能部署的开源/商业产品"定位是关键。

**风险（也见 §7.3）**：

- 恶意 / 无意误删 → 缓解：所有写操作走数据库，定期备份；UI 提供"撤销"
- 报告被错误验证后污染检索 → 缓解：UI 展示报告时显示"由谁验证、何时验证"，可以一键回退到 draft
- 自报身份可被冒用 → 缓解：内网物理隔离（`127.0.0.1` + 私有部署）使外部冒用不可能；UI 在身份显示处加"自报"标识让团队成员对此心知肚明；写操作内部记录 `client_id` 用于事后排查

#### 4.4.2 未来扩展 —— 鉴权对接接口

因为目标是**任意团队都能部署的开源 / 商业产品**，每家公司的员工账号体系不同（AD、LDAP、Google Workspace、自建 SSO、企业微信、飞书、钉钉……），未来加鉴权时**不能绑定特定供应商**。

**预留方向（PRD 不锁实现，只锁约束）**：

| 约束 | 含义 |
|---|---|
| 前后端各留扩展通道 | 后端：暴露 `AuthProvider` 抽象 + 请求中间件 hook 槽位；前端：暴露登录页 / 用户菜单 / 用户资料页等 UI 槽（slot） |
| 二开者不改核心代码 | 实现 Provider 接口即可（OIDC / LDAP / 企业微信 / 飞书 / 钉钉 / 自建 SSO 都是 `AuthProvider` 的不同实现） |
| 标准协议优先 | 第一批官方适配器至少覆盖 OIDC（OAuth2.1）、LDAP；其他通过协议适配 |
| 用户身份持久化 | 用户标识使用稳定的 `subject_id`（不依赖 email / 姓名） |
| 权限模型从简 | 第一版只有"成员 / 管理员"两级；多租户 / 多组织不在范围内 |
| 默认无鉴权可选 | 部署者可以选择"关闭鉴权"回到一期模式（自用、内网测试场景常用） |

这一节属于**扩展契约**——不在 MVP 范围，但 PRD 写进来是为了防止 MVP 阶段做出"无法演进"的设计决定。

## §5 成功指标

PRD 不锁具体数值（MVP 阶段拍数字没有意义，等真实部署校准），只锁**测量方法和方向**。

### 5.1 主指标 — Deflection Rate

> 提问者通过 CodeAsk 拿到答案、**不需要再去找真人**的比例。

**测量方式（双轨）**：

- **显式**：每个回答下方有"是否解决"按钮（解决 / 部分解决 / 没解决）—— 用户主动反馈
- **隐式**：回答后用户在 N 分钟内未产生新提问、未点击"找人帮忙" → 视为"自然结束"
- 两者相互校准：显式是金标准，隐式覆盖未反馈的场景

**方向**：

- MVP 上线第 1 个月：建立 baseline
- 第 3 个月：deflection rate 稳定上升趋势
- 长期目标：>50%（具体阈值随团队规模和知识库密度调整）

### 5.2 质量护栏 — 防止"高 deflection 但答错"

如果只追 deflection rate 不要质量护栏，**最坏情况是 Maintainer 比之前更累**（替系统纠错）。所以这一对必须共存。

| 护栏指标 | 测量 | 红线方向 |
|---|---|---|
| 错误反馈率 | 提问者主动 flag "回答错误"的比例 | 持续上升 = 警报 |
| 回流率 | 提问者拿到答案后，N 天内又来问"相同问题"的比例 | 高回流 = 实际没解决 |
| 报告审核拒绝率 | 报告草稿被审核拒绝的比例 | 高拒绝率 = Agent 推理质量差 |

### 5.3 飞轮健康指标（次级）

| 指标 | 解读 |
|---|---|
| Maintainer 月均审核通过的报告数 | 飞轮输入；为 0 = 飞轮没启动 |
| 已验证报告被命中次数 | 飞轮产出；持续为 0 = 报告库没价值或检索不命中 |
| 知识库文档新增 / 更新频率 | 知识库是否仍在被维护 |

### 5.4 反向指标（不该追的）

为了防止后续被误导，PRD 里明确列出**不追的指标**：

- ❌ "提问数量" —— 提问越多不代表越成功
- ❌ "Token 消耗" —— 花钱越多不代表越好
- ❌ "Agent 调用工具次数" —— 越多反而是回答效率低
- ❌ "回答字数" —— 长答不等于好答

## §6 Anti-Goal

明确**不做什么**。Anti-Goal 不是穷举所有"反对的事"，只列**最容易被误以为应该做、需要团队后续抵抗诱惑**的方向。

### 6.1 永久 Anti-Goal

| Anti-Goal | 原因 |
|---|---|
| 不做实时告警 / 监控 | 与 ELK / Prometheus / Sentry 不竞争；CodeAsk 是"分析"工具，不是"检测"工具 |
| 不做自动修 PR / 自动修复 | LLM 时代最容易被请求的方向，但和"不假装自信"承诺矛盾——错的 PR 比错的回答更难收拾 |
| 不替代真人 / 不假装自信 | 系统不确定就坦白；宁可 deflection 低，不出错答 |
| 不做无知识库的零样本通用代码助手 | 与 Copilot / Cursor / GitHub Copilot Chat 不在同一生态位 |
| 不做多租户 SaaS | CodeAsk 是私有部署产品，团队的代码和知识不能进多租户中心 |

### 6.2 MVP Anti-Goal（一期不做，未来可重新评估）

| Anti-Goal | 一期延后原因 |
|---|---|
| 不做日志平台直接对接 | 用户粘贴 / 上传足够；ELK / Loki / Splunk 对接复杂，每家不同 |
| 不做权限 / 审核工作流 | 已在 §4.4 决定；目标是降低部署门槛 |
| 不做向量检索作为核心依赖 | FTS5 + n-gram + 精确信号召回够用；向量后续可叠加 |
| 不做 IDE 插件 | Web 是更广覆盖的入口 |
| 不做客户端原生应用 | Web 足够 |
| 不做实时多人协同编辑 | 知识库非 Notion；编辑场景轻 |

## §7 关键假设与风险

按"如果错了，破坏多大"分级。**致命假设**是产品根基——错了就得推倒。

### 7.1 致命假设（错了 = 产品死）

#### A1：Maintainer 愿意持续维护知识库

| 项 | 内容 |
|---|---|
| 为什么致命 | 飞轮的输入端。Maintainer 觉得"维护时间 > 省下的被打扰时间" → 飞轮死 |
| 怎么验证 | alpha 用户跟踪两个数据 — "本月录入耗时" vs "本月被打扰减少时间" |
| 怎么缓解 | 把录入摩擦做到最低：拖拽目录 / 粘贴 / 链接外部 wiki / 从历史问答会话一键沉淀为知识库 |

#### A2：Agent 自动定界到特性足够准

| 项 | 内容 |
|---|---|
| 为什么致命 | 定界错 → 后面所有检索的特性范围都错 → 答案都是错的 |
| 怎么验证 | 建一个定界 eval 集（人工标注的"问题 → 正确特性"对），看 top-1 / top-3 准确率 |
| 怎么缓解 | 定界失败时 `ask_user`，让用户从候选特性列表中选；UI 醒目显示"Agent 选了哪个特性"，提问者可以一键改 |

#### A3：Agent 自动判断"知识库够不够"足够准

| 项 | 内容 |
|---|---|
| 为什么致命 | 判断过早进代码 = 慢、贵；过晚 = 用知识库残缺答案误导 |
| 怎么验证 | eval 集 + 用户反馈"答错了" vs "答得过浅"的分布 |
| 怎么缓解 | 让判断过程在 UI 透明（提问者看得到"已经查了知识库 3 篇文档但都不够"）；提供"再深查一下"按钮兜底 |

### 7.2 重要假设（错了 = 体验打折）

#### B1：提问者愿意接受"带证据的回答"

- **风险**：证据折叠不好 → 提问者觉得啰嗦 → 体验差 → deflection 不上升
- **怎么验证**：A/B 测试 "结论先 + 证据折叠" vs "全部展开"
- **怎么缓解**：UI 上结论放最显眼、证据默认折叠、可一键展开

#### B2：LLM 调用成本可承受

- **风险**：一次问答多次 LLM（定界、判断、合成、报告草稿）→ 团队舍不得用
- **怎么验证**：alpha 阶段跑真实数据，统计平均每次问答的 token / 成本
- **怎么缓解**：简单问答路径只调一次 LLM；支持本地模型 / OpenAI-compatible 自部署模型；知识库摘要级 prompt 缓存

#### B3：特性边界对真实代码有意义

- **风险**：用户实际的代码组织不是干净的"特性 → 仓库"映射，跨特性问题很多 → 定界无意义
- **怎么验证**：alpha 用户的真实特性配置形态——多少特性是 1 仓、多少是多仓、多少跨团队
- **怎么缓解**：允许特性 overlap；允许 Agent 跨特性检索（降级）；允许"无明确特性"的全局问答

### 7.3 次要假设（错了 = 局部修补）

| 假设 | 错了的影响 |
|---|---|
| 一期无权限模型不会被恶意滥用 | 私有部署内网通常 OK；恶意环境需要 §4.4.2 接入鉴权 |
| FTS5 + n-gram 检索质量够 | 不够时叠加向量检索，不需推翻现有 |
| 报告人工验证流程不会成为飞轮瓶颈 | 真成瓶颈时引入"自动入库 + 后置审核"模式 |

## §8 用户旅程

写 4 个真实场景（含失败路径），让 PRD 不停留在抽象。每个旅程标注它在验证哪些假设。

### 8.1 旅程 1 — Maintainer 初次部署到产生第一份价值

**角色**：Alice，订单团队 tech lead，被新人和兄弟团队反复问"订单流程"问题，决定试试 CodeAsk。

```text
Day 0  装好 CodeAsk（docker-compose / npm，30 分钟内）
       打开浏览器 → 默认无登录 → 直接进首页

Day 1  全局仓库池：填 order-service / payment-gateway 的 Git URL
       后端异步 clone（Alice 不等）→ 状态转为 ready

       建特性 "订单"：勾选关联 order-service + payment-gateway
       上传 4 份文档：《订单状态机》《下单接口》《支付回调》《灰度策略》

       Alice 自己测一个问题："订单失败后多久退款？"
       → Agent 定界到 "订单" → 检索知识库
       → 命中《支付回调》第 3 节 → 基于原文回答 + 引用 heading
       → Alice 觉得"够用"，关闭测试

Day 3  团队群里发链接："以后问订单的事先去 CodeAsk 试一下"
```

**这个旅程在验证**：A1（录入摩擦） / 4.4.1（无登录上手）。

**这个旅程容易卡的地方**：

- Git URL 需要 SSH key / Token → 全局仓库池配置页要支持凭证
- 文档拖拽必须保留目录结构，否则 heading path 全乱
- 异步 clone 失败时要有可见的错误提示，否则 Alice 无感

### 8.2 旅程 2 — Asker 简单问答 / 知识库够用 / deflection +1

**角色**：Bob，新加入订单团队 3 天，在写自己的第一个 ticket。

```text
Bob   打开 CodeAsk，新建会话
       提问："订单提交失败后有重试吗？"

Agent  定界 → 锁定 "订单" 特性
       UI 显示："正在查 4 篇相关文档..."
       命中《订单状态机》第 5 节 + 《支付回调》第 3 节
       判断"知识库够答" → 不进代码层
       生成回答："是。重试条件见《订单状态机》第 5 节，最多 3 次..."
       带证据：[订单状态机.md / 第 5 节 / 第 23-37 行]

Bob   点开证据快速核对 → 确认 → 点 "已解决"

Stats deflection rate +1
       Alice 没被打扰
```

**这个旅程在验证**：§3 主链路"知识库优先"路径走通 / B1 证据折叠接受度 / B2 简单问答成本可控。

### 8.3 旅程 3 — Asker 复杂问题 / Agent 答错 / 飞轮启动

**角色**：Carol，跨团队协作者，要改 `Order.payment_method` 字段，担心破坏下游。

```text
Carol  提问："如果在 order 表里把 payment_method 字段改成枚举，
              下游会有哪些代码受影响？"

Agent  定界 → 锁定 "订单" 特性
       检索知识库 → 没有"字段变更影响"类文档
       判断"知识库不够" → 进代码层
       UI 显示："正在分析 order-service / payment-gateway..."
       worktree 拷贝 → grep "payment_method" → 5 处直接引用
       read_file → 提取关键调用 → 合成回答

       回答："找到 5 处调用，列表..."
       带证据：5 个 file:line 引用 + commit
       置信度：medium（"可能存在反射或字符串拼接调用未覆盖"）

Carol  点 "部分解决" → 备注："漏了 settlement-service 里的反射调用"

Alice  收到通知（或在 dashboard 看到这条 medium-confidence 标"部分解决"）
       打开 → 看到漏的反射调用 → 决定沉淀知识
       一键创建文档草稿《订单字段变更影响清单》
       预填 Carol 的问题、Agent 的回答、Alice 的补充
       Alice 微调 5 分钟 → 保存到 "订单" 特性的知识库

2 周后 别人问类似问题 → Agent 直接命中这份新文档 → 不再进代码层
```

**这个旅程在验证**：A3 复杂度路由 / 失败模式可恢复 / 飞轮启动机制 / §6.1 不假装自信。

**这个旅程容易卡的地方**：

- Maintainer 看到"部分解决"信号需要主动行为；如果没有 dashboard 提醒，飞轮转不起来
- "一键沉淀"的录入摩擦如果太高（A1），Maintainer 不会做
- grep 处理不了反射调用是 Agent 的真实局限，UI 必须把"代码分析的覆盖范围"标清楚

### 8.4 旅程 4 — Oncall 救火 / 一期 CodeAsk 的真实定位

**角色**：Dave，凌晨 2 点被报警炸醒——告警："订单服务 5xx 飙升"。

```text
Dave   SSH 到生产 → 抓一段 error log → 复制
       打开 CodeAsk（旧的笔记本上一个 tab）→ 新建会话 → 粘贴日志
       附加现象描述："订单 5xx 飙升，最近 10 分钟"

日志关键信息：
  ERROR OrderService.submitOrder
  PaymentTimeoutException: timeout calling payment-gateway
  trace_id=abc123, ...

Agent  定界 → 锁定 "订单" 特性（错误信号匹配 OrderService + payment-gateway）
       检索知识库 → 命中《支付回调》—— 提到此异常但只讲业务层处理
       判断"知识库不够"（无 timeout 飙升排查类文档）→ 进代码层

       worktree 拷贝 order-service / payment-gateway
       grep PaymentTimeoutException 抛出点
       read_file PaymentClient.java → 看到 timeout=1s 是 hardcode
       git log 最近 7 天 → 这块代码没被改

       回答按"结论先 + 证据折叠"展示：
         结论（醒目展示）：
           可能根因 — payment-gateway 调用 timeout 是 hardcode 1s
           建议先操作：把 timeout 临时调大；联系 payment-gateway 团队看他们服务健康
         置信度 medium
         不确定点：
           - 没有 payment-gateway 服务自身的日志可看
           - 没看到流量 / QPS 数据，可能是流量突发或下游慢
         证据（折叠）：
           PaymentClient.java:45 / commit abc123
           《支付回调》第 3 节

Dave   照建议把 timeout 临时调到 3s → 发布 hotfix → 5xx 下降
       回 CodeAsk → 点 "已解决" → 备注 "临时止血，根因待查"

事后   Alice（白天上班看到 dashboard）→ 看到这条 oncall 会话被标"已解决" + 备注
       觉得是值得沉淀的案例 → 一键生成报告草稿
       Alice 补充：
         - 实际根因（payment-gateway 团队某 bug）
         - 长期方案（timeout 改成 config-driven，加 P99 报警）
       Alice 自己点 "验证通过"（一期无权限，谁都能验证）
       报告进入 reports_fts，标记高优先级

1 月后  另一个 oncall 粘贴类似日志 → Agent 直接命中这份已验证报告 → 30 秒拿到止血方案
```

**这个旅程在验证**：

- 主链路在 oncall 场景下能跑（不是 P0，但能用）
- "粘贴日志"作为输入足够，不需要日志平台对接（§6.2）
- Agent 给 medium 置信度 + 不确定点是诚实的（§6.1）
- 报告飞轮的真正价值：**oncall 第一次没那么神，但第二次同样事故时直接秒答**

**这个旅程暴露的产品要求**：

- UI 必须把"结论 + 建议操作"放最前面，证据默认折叠（凌晨 2 点没人读 5 屏证据）
- 回答**绝不能**写成"按这样改就行了"——必须明确是"建议操作"，决策权在人
- 报告生成 / 验证摩擦必须低（Alice 抽 5-10 分钟就能完成），否则飞轮断
- "找哪个团队帮忙"一期不内置，但知识库里如果有"团队 owner / 联系方式"文档，Agent 就能引用

**一期对 oncall 场景的诚实定位**：

> CodeAsk **不承诺**帮你解决 oncall 事故。它承诺：
>
> - 第一次事故：帮你**少走几步**（自动检索知识库、扫一眼相关代码、给出 medium 置信度的方向）
> - 第一次事故的副产品：留下一份可被沉淀为报告的会话记录
> - 第二次同类事故：在 30 秒内直接命中已验证报告 → 这才是真正的价值
